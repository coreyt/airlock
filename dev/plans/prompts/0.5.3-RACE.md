## Design Decisions (already resolved by the Phase-D design gate — you cannot infer these)

1. **The race fix is the PRIORITY.** `threat_detector.py` (~lines 123–124) does
   `combined = max(score, client.threat_score * decay_factor); client.threat_score
   = combined` — a read-modify-write **outside any lock**. Concurrent same-client
   requests can lose an update.
   - **Fix (codex-recommended):** expose a short mutator on the store —
     `StateStore.record_threat_score(...)` (name your call) — that, **under
     `StateStore._lock`**, snapshots whatever client state it needs (the request/
     error deque timestamps), computes the decay, applies `combined = max(score,
     prev * decay)`, writes `threat_score` **and** `backoff_until`, and returns the
     resulting assessment values. `assess_threat` computes the heuristic `score`
     **outside** the lock (the deque scans / text length / error-rate heuristics
     stay lock-free), then calls the mutator for the atomic RMW. **Keep logging
     outside the lock.**
   - **Behavior preservation:** for the single-threaded path the resulting
     `threat_score`, `blocked` decision, and `backoff_seconds` must be **identical**
     to today. Add a characterization test asserting that, plus the concurrency test
     below. Do NOT change thresholds, decay constant, or backoff formula.
2. **No-lost-update test.** Add a deterministic concurrent-same-client test
   (`tests/test_threat_score_race.py` or in `test_state_stress.py`) that, without
   the lock fix, could lose an update, and with it cannot — e.g. many threads/tasks
   each pushing a known score increment and asserting the final accumulated
   `threat_score` reflects **all** of them (no lost write). Make it deterministic
   (stress-loop with enough iterations, or injected ordering) — not flaky.
3. **Client-identity consolidation (one path).** There are 3 extraction paths:
   `airlock/fast/state.normalize_client_id`, the inline extraction in
   `airlock/callbacks/enterprise_logger.py`, and
   `airlock/client_identity.py` (`extract_airlock_client_from_headers`).
   Consolidate so there is **one** canonical implementation in
   `client_identity.py`, and have `guardian.py` (and `fast/state.normalize_client_id`
   if it duplicates logic) call it. **Semantics must not change** — the consolidated
   extractor must produce the **same `client_id`** for every existing input.
   - **Golden test (AC-RACE):** a parametrized test feeding a representative input
     set (headers/metadata variants incl. missing/empty/whitespace, the
     `x-airlock-client` header, fallbacks) and asserting the consolidated extractor's
     output equals what **all 3 legacy paths** produced. Capture the legacy outputs
     by reading the 3 implementations (do NOT guess) before you collapse them.
   - You may leave `enterprise_logger.py` calling its own path IF editing it is out
     of your ownership — instead, make `client_identity.py` the canonical impl and
     point `state.normalize_client_id` + `guardian.py` at it; the golden test proves
     `enterprise_logger`'s behavior matches. (Editing enterprise_logger is NOT in
     your MODIFY set — if true consolidation requires changing it, STOP and escalate;
     the safe in-scope move is: canonical impl in client_identity, callers in your
     MODIFY set routed to it, golden test proving parity with enterprise_logger.)
4. **Config-loading consolidation is OPTIONAL and LOWEST priority.** The plan notes
   "if consolidation balloons, split it out and ship the race fix alone." Do the
   race fix + identity consolidation. **Only** consolidate config loading if it is a
   small, obvious helper extraction touching your MODIFY files; otherwise SKIP it,
