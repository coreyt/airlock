## Design Decisions (already resolved — you cannot infer these)

1. **Threshold, not one-strike (A1).** Quarantine a client→provider pair only after
   `rate_limit_threshold` 429s within `rate_limit_window_seconds`. **Default
   threshold = 1 preserves today's behavior** (CC-3). The 429 is always *recorded*
   in the deque; only the *quarantine arming* is threshold-gated.
2. **Per-client policy (E).** A `BreakerPolicy` (frozen dataclass) keyed by
   `client_id` supplies `{rate_limit_threshold, rate_limit_window_seconds,
   client_cooldown_seconds, provider_cooldown_seconds,
   provider_escalation_client_threshold, escalation_exempt, disabled}`. Add a
   `BreakerConfig` holder on the `store` with `policy_for(client_id) -> BreakerPolicy`.
   Precedence: **per-client override → global default → hard-coded constant** (CC-2).
3. **`cleared_at: float = 0.0` (CC-6) — YOU OWN IT.** Add to **both**
   `ClientProviderState` and `ProviderState`. Apply the floor to **every reader of
   rate-limit history that can re-arm a breaker**:
   - `recent_rate_limit_count(window)` counts only events with
     `t > max(now - window, cleared_at)`.
   - **`ProviderState.impacted_clients()` must apply the same floor** (per-bucket
     and provider-level `cleared_at`) — otherwise a future provider clear re-arms on
     pre-clear client history. (This was a codex BLOCK; do not skip it.)
   You do NOT add the admin clear method; you DO add the floored readers and a unit
   test that sets `cleared_at` directly and proves the floor works.
4. **`_half_open_probe: bool = False` (CC-7) — YOU OWN IT.** Add to both
   `ClientProviderState` and `ProviderState` (today only `ModelState` has a circuit;
   mirror its half-open recovery at `airlock/fast/state.py` ~421/435). Add the
   transition logic: when `_half_open_probe` is set, the next outcome resolves it —
   a success clears `_half_open_probe` + `quarantine_until` (closed); a 429 re-arms
   `quarantine_until` via the policy cooldown. You do NOT add the method that *sets*
   `_half_open_probe` (that's the admin probe-clear, Pack ADM-state); test the
   transition by setting the flag directly.
5. **`escalation_exempt` clients do not count toward provider-wide escalation** —
   fixes the "2 clients → everyone blocked" blast radius. `disabled` policy → the
   breaker never arms from 429s and the guardian pre-call skips the quarantine raise
   for that client.
6. **Cooldown still `max(quarantine_until, now + cooldown)`** (no additive stacking);
   threshold gating happens *before* arming. Use `policy.client_cooldown_seconds` /
   `policy.provider_cooldown_seconds`.
7. **No-re-arm invariant (A1-b).** Confirmed not currently possible (pre-call raises
   route through `post_call_failure_hook`, not the breaker-feeding
   `log_failure_event`). Add an explicit regression test asserting a pre-call
   quarantine block does **not** call `record_provider_rate_limit` (mock the store
   method, assert `call_count == 0`), so a future refactor can't silently introduce
   the loop.
8. **Startup-read config (CC-2).** Parse `airlock_settings.circuit_breaker` +
   `AIRLOCK_BREAKER_OVERRIDES` (JSON) once at startup; inject into `store`. No
   hot-reload. Malformed env JSON → fall back to defaults with a logged warning
   (do NOT crash startup).
9. **CC-3 invariant:** a config-free deploy behaves EXACTLY as today (threshold 1,
   300 s cooldowns, escalation 2). This is the most important acceptance check.
