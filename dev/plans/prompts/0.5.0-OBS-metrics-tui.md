## Design Decisions (resolved — do not redesign)
1. **Counter (CC-T1 one ledger).** In `_build_metrics`, add:
   ```python
   "mutations_total": Counter("airlock_mutations_total",
       "Total ledger mutations by field and operation", ["field", "op"]),
   ```
   Increment in `AirlockMetricsCallback.log_success_event` (and `log_failure_event` if a
   ledger is present there) by iterating `metadata.get("airlock_mutations") or []`:
   `_metrics["mutations_total"].labels(field=m.field, op=m.op).inc()`. The ledger holds
   `Mutation` objects with `.field`/`.op`. Guard against a non-iterable/empty ledger.
   Watch label cardinality — `field`/`op` are bounded (op is a 7-value enum; field is a
   small fixed set of request keys), so this is safe.
2. **Register the callback.** `AirlockMetricsCallback` is defined but not wired. Add a
   `_self_register()` (mirroring `monitor.py:385-399` / `enterprise_logger.py:546-555`) so it
   joins litellm's success/failure callback lists. Without this the counter never increments
   in production. Keep registration idempotent (guard against double-add), matching the
   existing callbacks' pattern.
3. **`classify_backend_kind` helper (additive).** Extract the existing provider→backend_kind
   logic from `attribute_served_backend` into a public module-level helper in
   `transparency.py`:
   ```python
   def classify_backend_kind(provider: str | None) -> Literal["native", "gateway", "unknown"]:
       # gateway: bedrock/azure/vertex_ai ; native: anthropic/openai/gemini ; else unknown
   ```
   Refactor `attribute_served_backend` to call it (same behavior — keep all existing tests
   green). This is the ONLY change to `transparency.py`'s surface (purely additive).
4. **TUI "Served via" column.** backend_kind is a pure function of the provider NAME, and each
   TUI provider row already has the provider name (from `store.all_providers()`). Add a
   "Served via" column whose cell = `classify_backend_kind(provider_name)` (render "native"/
   "gateway"/"unknown" or a short glyph). Append the column in `add_columns` (~`:193`) and the
   value to each row tuple (~`:644`) and the render (~`:734`). No new per-request state, no
   store changes. Observe-only.
