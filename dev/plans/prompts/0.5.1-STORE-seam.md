## The problem this pack fixes (two coupled bugs)

1. **R5 undercount.** `ProviderSpend.spend_records = deque(maxlen=1000)`
   (`state.py:255–267`) is summed over a trailing 24h window. A provider doing
   >1000 billed calls/day drops in-window records off the deque → daily spend
   **undercounts exactly on the busy providers budgets protect**. It also feeds the
   advisor `daily_spend_usd` (`advisor/tools.py:122`).
2. **FIX-1 process-location bug.** `restore_state`/`checkpoint_state` (`state.py:1290`,
   `:1317`) are wired in `proxy.py` — `restore_state` at `:437–444` in `main()` and
   `checkpoint_state` at `:306–313` in the SIGTERM handler — i.e. the **launcher**
   process. But the `store` is mutated only inside the **litellm subprocess**
   (`proxy.py:450 subprocess.run(litellm_cmd)`), where `airlock.fast.monitor`'s
   `log_success_event` runs as a callback and records spend at `monitor.py:224`. The
   launcher and child are **separate processes with separate memory** — so the
   launcher checkpoints an *empty* store and the child's state is never persisted.
   This already silently defeats the existing circuit-breaker checkpoint.

## Design decisions (already resolved — do not re-litigate; HITL-confirmed)

- **In-memory, single-process only. NO Redis, NO `--num_workers`.** (Q2: multi-worker
  deferred.) Build the seam so a future Redis backend is a config flip, not a rewrite,
  but do not add Redis now.
- **Q3: a restart must NOT reset captured spend.** Persist Airlock provider spend to
  disk via a file checkpoint and rehydrate in-window spend on restart. Scope is
  **Airlock's warn/swap spend only** (FIX-2) — do NOT try to restore LiteLLM's own
  `provider_spend:{provider}:{duration}` hard-budget cache (moot today, all budgets 0).
- **Rolling window, NOT tumbling.** Keep a trailing-window accumulator (timestamped
  buckets pruned by age). A tumbling daily-reset key is explicitly REJECTED (it would
  zero spend at rollover, conflicting with Q3).
- **Integer µ$ (FIX-4).** Accumulate spend as integer **micro-dollars** (`int(round(
  cost_usd * 1_000_000))`), not float increments, so sync/Redis `increment_cache`
  (integer `INCR`) and the in-memory backend agree. `recent_spend()` keeps returning
  **float USD** (sum of µ$ / 1e6) so `advisor/tools.py:122` and the monitor are
  unchanged at the call site.
- **Airlock-owned cache config, not DualCache defaults (FIX-3).** A default in-memory
  `DualCache` carries a ~10-min TTL (`default_ttl or 600`) and no locking. For a 24h
  window that would expire spend early and lose concurrent increments. Set an
  **explicit per-key TTL ≥ the rolling window** on every spend key (do NOT rely on
  `default_ttl=None`), use an adequate max size, and hold the existing
  `StateStore._lock` (`state.py:685`, an `RLock`) around compound read-modify-write.

## Suggested approach (you may adapt the mechanism; the DoD properties are the bar)

Back the per-provider spend with **time-bucketed integer-µ$ counters** in a
`DualCache` (e.g. one key per provider per fixed-width bucket, bucket width ≤ 1h):
`record_spend` increments the current bucket's key by `µ$`; `recent_spend(window)`
sums the buckets whose bucket-start is within `now - window`; old buckets are pruned
by age (and naturally expire via the per-key TTL ≥ window). This removes the
per-record cap (fixes R5 — accuracy is independent of call volume) and maps cleanly
to `increment_cache`/Redis later. Abstract this behind an Airlock-owned interface in
`state.py` (e.g. a `SpendStore`/store method surface) so swapping the in-memory
`DualCache` for Redis later touches one module. Preserve the public API used by
callers: `store.get_provider_spend(provider).record_spend(now, cost)` and
`.recent_spend()` must keep working (refactor the internals, keep the seam).

### Checkpoint semantics (FIX-7 — all required)

- **Versioned schema** (include a `version` field).
- **Atomic write**: write a temp file then `os.replace` onto the target.
- **Prune-before-checkpoint**: drop out-of-window records/buckets before writing.
- **Idempotent, replace-not-append restore**: rehydrating must not double-count if a
  record/bucket is already present; restore is **bounded by record/bucket age** (only
  in-window data rehydrated) — NOT by the 5-minute `cb_state` freshness gate, which is
  a breaker-specific staleness guard and must remain **breaker-only**
  (`_CB_STATE_MAX_AGE_SECONDS`, `state.py:1287`/`:1331`).

### FIX-1 — checkpoint/restore must run in the litellm child process

The `airlock.fast.monitor` callback runs inside the litellm subprocess. Move/duplicate
the spend (and the existing breaker) checkpoint+restore into **that** process:
**restore on the child's startup** (when the monitor initializes in the child) and
**checkpoint periodically and on the child's shutdown**. The launcher's current
`restore_state`/`checkpoint_state` calls in `proxy.py` operate on the wrong (empty)
store — remove or neutralize them so the child is the single source of truth (do not
leave a second writer racing the child on `cb_state.json`). Keep the existing
`cb_state.json` path/filename for breaker recovery; the spend checkpoint may share the
file (extend the versioned schema) or use a sibling file in the same `AIRLOCK_STATE_DIR`
— your call, but **breaker recovery must still round-trip** and the launcher must not
also write it.
