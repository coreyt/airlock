### Required test classes/assertions

1. **Old↔new alias PARITY (no client breaks).** For each consolidated/renamed family,
   the legacy alias and its new `provider/model` alias must agree on resolution target +
   provider attribution:
   - `infer_provider("aistudio/gemini-3.5-flash") == infer_provider("gemini-3.5-flash-aistudio") == "gemini"`;
     `infer_provider("vertex/gemini-3.5-flash") == infer_provider("gemini-3.5-flash-vertex") == "vertex_ai"`;
     `infer_provider("mistral/mistral-large") == infer_provider("mistral-large-batch") == "mistral"`;
     `infer_provider("vllm/qwen3.6-27b") == infer_provider("qwen36-27b-vllm-batch") == "openai"`;
     `infer_provider("anthropic/claude-haiku") == infer_provider("claude-haiku") == "anthropic"`.
   - Every legacy alias still `resolve()`s to a real `model_name` (none returns None).
   - (call `set_router_config(cfg)` and `alias_table.load_from_config("config.yaml")` first.)

2. **Collision-safety on the REAL catalog.** `resolve("gemini-3.5-flash")` → the bare
   AI-Studio entry (never `vertex/…`/`aistudio/…`); `resolve("vertex/gemini-3.5-flash")`
   and `resolve("vertex_ai/gemini-3.5-flash")` → the vertex entry;
   `resolve("aistudio/gemini-3.5-flash")` and `resolve("gemini/gemini-3.5-flash")` → an
   AI-Studio entry; `resolve("gemini-3.1-pro-preview")` (ambiguous body, both providers,
   no explicit alias) → `None`.

3. **Batch backend PARITY (old + new alias → same backend).** `load_batch_aliases(cfg)`
   contains BOTH the new consolidated keys and the legacy twins, each with the right
   `backend`/`provider_model`, for all three gateway families:
   - `aistudio/gemini-3.5-flash` and `gemini-3.5-flash-aistudio` → `backend == "aistudio"`,
     same `provider_model`; `aistudio/gemini-3.1-pro` and `gemini-3.1-pro-aistudio`;
   - `mistral/mistral-large` and `mistral-large-batch` → `backend == "mistral"`; `mistral/mistral-small` and `mistral-small-batch`;
   - `vllm/qwen3.6-27b` and `qwen36-27b-vllm-batch` → `backend == "vllm"`.
   Also: bare sync entries (e.g. `gemini-3.5-flash`, `mistral-large`, `qwen3.6-27b`) are
   NOT in `load_batch_aliases` (no marker → sync-only).

4. **Capability ↔ wiring consistency for EVERY entry.** For every `model_list` entry:
   `"batch" in endpoints_for(entry)` **iff** (truthy `entry.get("airlock_batch")` OR
   (`litellm_params.model` startswith `vertex_ai/` AND `vertex_location` regional)).
   In particular assert the vertex-at-`global` entries are `["chat"]` (no over-claim) and
   every `airlock_batch`-marked entry is `["chat","batch"]`. No entry over/under-claims.

5. **The two capability surfaces AGREE (single source).** For representative entries, the
   `model_info` that `_prepare_runtime_config` injects (parse the returned temp config) and
   the `airlock` object the `/v1/models` seam map (`_build_capability_map()`) produce are
   BOTH equal to `capability_record(entry)`. (Lock that /model/info and /v1/models can't drift.)

6. **`/v1/models` seam is additive + non-breaking** (integration, in-process): drive
   `ModelsCapabilityMiddleware` over a canned OpenAI `/v1/models` response (ids from the
   real catalog) via `httpx.ASGITransport`; assert each model keeps `id`/`object` and gains
   a correct `airlock` object; a non-`/v1/models` path passes through unchanged. (If
   CAP-v1models' own tests already cover this thoroughly, keep your version a thin
   integration check using REAL catalog ids — don't duplicate the unit edge-cases.)

7. **Reference integrity (stale-target guard).** Every `router_settings.fallbacks` target
   and every `cost_tiers` member is a live `model_name` in `model_list`.
