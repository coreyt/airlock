## Design Decisions (already resolved by the Phase-D design gate — you cannot infer these)

1. **Presidio offload is behavior-preserving — only threading changes.** Wrap the
   synchronous scrub work in `await asyncio.to_thread(...)` inside
   `async_pre_call_hook` (e.g. offload the `_scrub_messages` / `_scrub_mcp_arguments`
   calls). The **redaction output, the mapping, and the counters must be
   byte-identical** to today (dict mutation inside the thread is fine under the GIL;
   keep the request-local `mapping`/`counters` ordering). No change to entities,
   thresholds, or the redaction format. Tests assert byte-identical redaction +
   mapping before vs after.
2. **Concurrency win (UN-27).** Add a test proving N concurrent `AIRLOCK_PII_ENABLED`
   requests no longer serialize: either assert wall-clock ≪ N × single-request
   latency, OR an event-loop-not-blocked interleaving probe (e.g. a concurrent
   `asyncio.sleep`-based heartbeat that keeps ticking while Presidio runs). Make it
   robust, not flaky (use a fake/slow analyzer or monkeypatch `analyzer.analyze` with
   a `time.sleep` so the offload is observable deterministically).
3. **Per-request text cache, metadata-scoped, POST-redaction.** `extract_text(data,
   call_type)` becomes cache-aware: if `data["metadata"]["_airlock_text"]` is present,
   return it; else compute from the **current** messages and store it there. New
   request = new `data` = fresh cache (no cross-request leakage).
   - **Order correctness (THE subtlety — document it):** the cache must reflect the
     **post-PII-redaction** text. Because pii runs first and is the only pre_call guard
     that mutates messages, populate/refresh the cache **at the end of the pii stage
     after redaction** (`pii_guard` sets `data["metadata"]["_airlock_text"]` to the
     freshly extracted post-redaction text). When PII is disabled, pii returns early
     without setting it and the first downstream `extract_text` caller (keyword)
     computes + caches from the unredacted (= unchanged) messages. Either way keyword
     + guardian see the correct current text and extraction runs **once**.
   - Guard against staleness: if any code mutates `data["messages"]` it must invalidate
     the cache; in this pack only pii mutates, and it refreshes the cache itself.
   - Pin the order in a test: with PII enabled, the value keyword/guardian receive via
     `extract_text` equals the **redacted** text (not the original), and `extract_text`
     is invoked once per request (assert via a call counter / spy).
4. **vLLM TTL widen + prewarm (opt-in safe).** Change `_DEFAULT_TTL` 5.0 → 30.0 (and
   the docstring/env-default mentions of "5s" → "30s"). Add a **prewarm** that
   populates the `/models` cache on startup (best-effort; failure must be swallowed —
   a cold cache MUST still resolve correctly on the first request, and non-vLLM
   aliases that never hit this path must be unaffected). If a clean startup hook isn't
   available without touching a DO NOT TOUCH file, implement the prewarm as an
   idempotent helper the router can call lazily and note the wiring in `output.json`
   rather than forcing a risky edit — escalate if unsure.
