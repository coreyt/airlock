## Design Decisions (resolved — do not redesign)
1. **Three additive fields** in the record dict returned by `_build_record`, before the
   `airlock_client` assignment (~`:510`):
   - `mutations`: the full ledger = `metadata.get("airlock_mutations") or []`, each
     `Mutation` serialized to a plain dict (use `dataclasses.asdict(m)` — or the module's
     `_serialize` if a value isn't JSON-native). This is the FULL ledger (NOT byte-bounded
     — that bound is only for the header). It is safe to log in full: redact records are
     value-free by construction and inject records carry `after=None` (OBS-ledger/CC-T2),
     so no secret/prompt text is present. Keep ledger order.
   - `served`: `attribute_served_backend(response_obj, cost_fallback=kwargs.get("response_cost"))`
     serialized to a dict (`dataclasses.asdict` on the `ServedBackend`), or `None` when the
     helper returns `None`. **Call it here directly** (Decision 5) — do NOT read a stashed
     `metadata["airlock_served"]`; the success hook is authoritative and `response_obj` here
     is the assembled response with final `_hidden_params`/`response_cost`.
   - `attribution`: `"served"` when the `ServedBackend` is non-None AND `served.provider`
     is not None; else `"inferred"`.
2. **`airlock_provider` (inferred) stays** exactly as-is (`:437`) — `served` sits beside it;
   the two are explicitly distinct (CC-T3). `record_type` stays `"request"` (no schema break).
3. **Self-sufficient (Decision 5):** the fields must be produced even when the response-headers
   hook never ran (e.g. a code path that skips it) — because attribution happens here.
4. **No behavior change** beyond the additive fields. Don't reorder/rename existing keys.
