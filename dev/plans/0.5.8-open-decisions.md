### Findings that emerged from the live verification

1. **Bare `gpt-5.6` is not a real OpenAI model id.** Only sol/terra/luna are offered.
   litellm's map carries a bare entry, which is what misled the earlier analysis. This
   **validates** pinning the convenience alias to the explicit `-sol` body — the
   floating id would have 404'd every request. Design §2.2, risk 17.

2. **D-5 nuance worth a decision.** The template previously pointed `gemini-flash-lite`
   at `gemini-3.1-flash-lite`, which **is real**. Aligning it to root moved it back to
   `gemini-2.5-flash-lite` — correct for parity, but a *downgrade* from a working newer
   model. Separately, **root's own generic Gemini aliases may be stale**: root pins
   `gemini-pro` to 2.5 while exposing 3.x only under `-preview` names. Whether the
   generic aliases should advance a generation is unexamined and should not be settled
   by accident.

3. **D-1's blast radius extends past GPT-5.6.** Strict resolution 404s *any* cross-tier
   fuzzy match that silently worked before. That is the intent, but the affected set is
   unknown for the same reason as the `reasoning_effort` change — hence both share the
   P-6c warn-only window. Risk 16.

## D-1 🔴 — Fuzzy alias resolution vs. "don't substitute silently"

**The tension.** In this session you established a principle for `reasoning_effort`:
*Airlock should not muck around; give clients what they ask for, and reject what's
invalid with a useful error.* Airlock's fuzzy model-alias resolver does the **opposite**
— it silently substitutes a different model whenever a name scores above 0.50.

That principle and that feature cannot both stand unmodified. GPT-5.6 makes the
conflict concrete: `gpt-5.6-mini` / `-nano` / `-pro` don't exist, score 0.57–0.59, and
silently resolve to **Sol at $5/$30** — logged at DEBUG. Someone reaching for "the
cheap 5.6" is billed 5× and never told.

| # | option | effect |
|---|---|---|
| A | Guess-guard aliases (map `-mini`/`-nano`→luna, `-pro`→sol) | Wrong guess lands cheap. But still silent substitution, and invents model names that leak into `/v1/models` — needs an `airlock_synthetic` marker. |
| B | Keep fuzzy routing, force WARNING on the misroute | Honest logging, no invented names. Cost surprise still happens; nobody reads DEBUG *or* WARNING in time. |
| C | Reject any non-exact match with a "did you mean" 400 | Fully consistent with the `reasoning_effort` principle. But kills a feature with real value — typo tolerance, and dated snapshots (`gpt-5.6-sol-2026-07-09` → `gpt-5.6-sol`). Broad blast radius. |
| **D** | **Auto-route only within the same cost tier; reject across tiers with "did you mean"** | **Recommended.** |

**Recommendation: D.**

**Justification.** The fuzzy resolver's value is real but is entirely about *convenience*
(typos, version suffixes). Its harm is entirely about *money* — silently serving a model
5× more expensive than the one requested. Gating on cost tier keeps the convenience
exactly where it is harmless and applies your principle exactly where it matters. It
also needs no invented model names, so the `/v1/models` truth problem in option A
disappears, along with the `airlock_synthetic` plumbing.

Concretely: `gpt-5.6-sol-2026-07-09` → `gpt-5.6-sol` still auto-routes (same tier).
`gpt-5.6-mini` → would cross low→high, so it returns:

```
400  Unknown model 'gpt-5.6-mini'. Did you mean 'gpt-5.6-luna'?
     Available: gpt-5.6-sol, gpt-5.6-terra, gpt-5.6-luna.
```

**Note:** D supersedes your earlier "map guesses to sensible targets" choice, which was
made before the discovery/attribution consequences were understood. If you prefer to
hold that choice, option A still works — it just carries the synthetic-marker work and
the silent-substitution inconsistency.

## D-2 🔴 — Is `max` a valid reasoning_effort for GPT-5.6?

OpenAI's docs list `none/low/medium/high/xhigh/max`. **litellm v1.93.0 sets no
`supports_max_reasoning_effort` key for any 5.6 entry.** Under the new strict-validation
contract, this single bit decides whether `max` is accepted or 400s.

| # | option | risk |
|---|---|---|
| A | Trust litellm; reject `max` | Rejects a level OpenAI documents as valid → breaks legitimate clients. |
| B | Allow-list `max` pending confirmation | Accepts a level the API may 400 on → error surfaces from upstream, confusingly. |
| **C** | **One live API call to settle it, then A or B** | **Recommended.** |

**Recommendation: C.**

**Justification.** This is the exact thing your item-2 principle forbids guessing about.
Rejecting a valid level and accepting an invalid one are *both* failures, and we
currently cannot tell which we'd be committing. The cost is one minimal completion
request against `gpt-5.6-sol` with `reasoning_effort: "max"` — cents. **I have not run
it: it spends against your account and needs your key.** Say the word, or run it
yourself and paste the result.

## D-3 🟡 — litellm over-counts GPT-5.6 long-context spend

GPT-5.6 is **flat-rate** — no long-context surcharge (design §12.4). But litellm v1.93.0
still carries `input_cost_per_token_above_272k_tokens` for all four 5.6 entries, so any
request over 272K input tokens is costed at **2× input / 1.5× output**. Harmless today
(5.6 isn't in config); **live the moment the 5.6 entries land.**

| # | option | effect |
|---|---|---|
| A | Accept, file upstream | Budgets trip at ~half the real spend on long-context 5.6 — and 1.05M context is the headline feature, so those requests are the *expected* case. |
| B | Override the 5.6 prices in `config.yaml` (litellm supports per-model cost overrides) | Correct accounting immediately; a local override to remove when upstream fixes it. |
| **C** | **B + A: override locally, file upstream, remove the override when fixed** | **Recommended.** |

**Recommendation: C.**

**Justification.** Over-counting is not a cosmetic reporting error — provider budgets
and near-limit downgrade routing *enforce* on these numbers, so a 2× error throttles
traffic at half the intended spend. The override is a few lines of config with a dated
comment. Filing upstream is what actually retires the workaround.

## D-5 🟡 — RATIFY OR REVERSE: Gemini template aliases (decided unilaterally)

**I changed this without asking.** The template pointed `gemini-pro` →
`gemini/gemini-3.1-pro` and `gemini-flash` → `gemini/gemini-3-flash`, **neither of which
exists** (only `-preview` variants do). I aligned all three generic Gemini aliases to
root's stable 2.5 line, because the new parity test demands template and root agree and

| # | option | |
|---|---|---|
| **A** | **Keep the alignment (2.5 stable line)** | **Recommended.** |
| B | Point generic aliases at the 3.x `-preview` ids | Newer, but preview ids move and can vanish; generic aliases are the worst place for that. |

**Recommendation: A**, with a follow-up: **root's own `gemini-pro` → `gemini-2.5-pro`

the generic Gemini aliases should advance a generation is a real question this work
didn't examine and shouldn't decide by accident.

## D-6 🟡 — Rollout of the `reasoning_effort` breaking change

Under strict validation, `none`/`off` sent to older OpenAI models becomes a **400**
where it previously became a silent `minimal`. Clients depending on it don't know they
are — the translation was never visible.

| # | option | |
|---|---|---|
| A | Ship the break in 0.5.7 | Correct immediately; some clients break with no warning. |
| **B** | **One release warn-only: log + response header, don't reject; enforce next release** | **Recommended.** |
| C | Opt-in flag, old behaviour default | Indefinitely preserves the bug; nobody migrates. |

**Recommendation: B.**

**Justification.** The failure mode this fixes is *silent* — which means the affected
population is also silent, and cannot be enumerated from here. A warn-only release turns
that unknown set into a measured one before anything breaks. Cost is one release cycle;
the alternative is discovering the blast radius from support tickets. This aligns with
the reason we're making the change at all: surface what was hidden.

## D-8 🟢 — Release numbering and what goes in the hotfix

Currently on branch `fix/model-config-hardening` with **two** logically separate changes:
the template config fix (urgent, data-only) and the `_apply_cost_tier` alias-equivalence
fix (behaviour change + tests).

| # | option | |
|---|---|---|
| **A** | ~~0.5.5.1 = template config only~~ — **SUPERSEDED**, not valid semver. Actual outcome: hotfix = **0.5.6**, admission gate → 0.5.7, GPT-5.6 work → 0.5.8. | |
| B | 0.5.5.1 = both | Simpler, but puts a behaviour change in an urgent patch. |
| C | Renumber: this work → 0.5.6, admission gate → 0.5.7 | Disturbs a committed plan for no gain. |

**Recommendation: A.**

**Justification.** An urgent hotfix should be the smallest reviewable thing that stops
the bleeding. The template fix is pure data with test coverage and near-zero risk; the
router change alters routing behaviour and deserves a normal review cycle. Splitting
costs one extra branch.

## D-9 🟢 — Does `X-Airlock-Model-Alias` ship in 0.5.7?

The header (plus its `model_successors` config block, loader, validation, and tests) is
the **largest net-new surface** in the plan. It exists only because `gpt-5` deliberately
stays on 5.5 and clients need telling that 5.6 exists.

| # | option | |
|---|---|---|
| **A** | **Ship in 0.5.7, sequenced last and explicitly cuttable** | **Recommended.** |
| B | Defer to 0.5.8 | 0.5.7 then ships 5.6 with no in-band discovery — the gap your D1 choice was meant to close. |

**Recommendation: A.**

**Justification.** It's the payoff for keeping `gpt-5` on 5.5: without it, that decision
just means users silently stay a generation behind. Sequencing it last means a
schedule squeeze drops it cleanly rather than forcing a rushed job.

## D-10 🟢 — Batch endpoint support for GPT-5.6

All three variants advertise `/v1/batch`, but Airlock only claims batch on an explicit
`airlock_batch` marker. Design §8.3 omits it.

**Recommendation: confirm the deferral.** Claiming batch without an exercised round-trip
would violate the anti-overclaim rule in `capability.py:72`. Overclaiming is worse than
underclaiming — a client that trusts `/model/info` and builds a batch pipeline on it
fails at runtime.
