## Verdict: BLOCK

**BLOCK**
- BLOCK CC-8/UN-10: The admin API has no explicit client→provider quarantine clear, but current pinned requests block on `ClientProviderState.quarantine_until` before provider-wide state, so clearing only `/providers/{p}` can leave the client still blocked; add `clear_client_provider_quarantine(client_id, provider, ...)` or define provider clear to clear all matching client-provider buckets. Verified at [state.py](/home/coreyt/projects/airlock/airlock/fast/state.py:165), [guardian.py](/home/coreyt/projects/airlock/airlock/fast/guardian.py:224), umbrella CC-8 [design-resilience-and-admin-overview.md](/home/coreyt/projects/airlock/dev/notes/design-resilience-and-admin-overview.md:66), admin ops [design-admin-api-capability-auth.md](/home/coreyt/projects/airlock/dev/notes/design-admin-api-capability-auth.md:253).
- BLOCK CC-11/security: `sub == client_id` does not prevent replay if `client_id` is resolved from forgeable `X-Airlock-Client`; current guardian/monitor prefer that header, and the admin note acknowledges it is unauthenticated. For guardrail-skip PDP, bind `sub` to authenticated key-derived identity, not attribution header. Verified at [guardian.py](/home/coreyt/projects/airlock/airlock/fast/guardian.py:66), [monitor.py](/home/coreyt/projects/airlock/airlock/fast/monitor.py:41), [client_identity.py](/home/coreyt/projects/airlock/airlock/client_identity.py:33), [design-admin-api-capability-auth.md](/home/coreyt/projects/airlock/dev/notes/design-admin-api-capability-auth.md:55).
- BLOCK CC-6: The cleared floor is specified for `recent_rate_limit_count`, but provider-wide escalation currently uses `ProviderState.impacted_clients()`; unless `impacted_clients()` also floors on `cleared_at`, a provider clear can re-arm on pre-clear client history after the next 429. Verified at [state.py](/home/coreyt/projects/airlock/airlock/fast/state.py:287) and [state.py](/home/coreyt/projects/airlock/airlock/fast/state.py:618).
