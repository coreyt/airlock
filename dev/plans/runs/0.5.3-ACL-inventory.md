> Read-only enumeration of every read of LiteLLM internals in `airlock/`,
> taken at `main` = HEAD `fc67c33` (v0.5.2). The ACL (`airlock/litellm_adapter.py`)
> must cover **all** of these or it is not an ACL. Test files excluded.
> Re-grep before the ACL implementer relies on any anchor — line numbers drift.

## Categories of "internal read"

LiteLLM-internal surfaces Airlock reaches into (undocumented / version-coupled,
pinned to 1.89.0 behavior under a wide `litellm[proxy]>=1.83.4,<2` range):

1. **`response._hidden_params`** — the per-response private dict (served-by,
   provider, cost, additional_headers).
2. **`response.custom_llm_provider`** — wrapper attribute (streaming wrapper
   hardcodes `vertex_ai_beta`; see `transparency.py:60`).
3. **`response_cost`** — both from `_hidden_params` and from the `CustomLogger`
   `kwargs` dict.
4. **`additional_headers`** — nested inside `_hidden_params` (read on success,
   written on the override path).
5. **proxy-app object** — `sys.modules.get("litellm.proxy.proxy_server")` +
   poking `app.middleware_stack` / `app.state` for middleware injection.

## ACL boundary questions for the design gate (Phase D)

- **Q1 — `kwargs.get("response_cost")` from the `CustomLogger` hook.** This is the
  *documented* logging-callback kwargs dict, not `_hidden_params`. Does the ACL own
  reads of the logging `kwargs` contract too, or only the truly-private
  `_hidden_params`/wrapper-attr surface? Recommendation: ACL owns a
  `served_backend_from_response(response, cost_fallback=...)` accessor (covers
  `transparency`/`monitor`), but leave bare `kwargs.get("response_cost")` reads in
  the loggers as the documented surface — UNLESS the gate wants one accessor for
  cost too. Pin this.
- **Q2 — middleware *install* vs app *resolution*.** The ACL clearly owns
  `resolve_proxy_app()` (the `sys.modules.get(...)` + `app.middleware_stack` access).
  Does it also own the install helper (the `middleware_stack is None ? add_middleware
  : wrap`) or does that stay in each caller / move to `proxy_bootstrap.py` (DECOUPLE)?
  Recommendation: ACL exposes `resolve_proxy_app()` and a generic
  `install_asgi_middleware(app, factory)` that encodes the started/not-started
  branch once; callers (batch/admin/models_seam) pass their middleware factory.
  `proxy_bootstrap.py` (DECOUPLE) orchestrates *order*; the ACL owns the *mechanism*.
- **Q3 — `batch/middleware.py` `custom_llm_provider` query param.** This is a
  client-supplied ASGI query param, NOT a LiteLLM internal — explicitly OUT of ACL
  scope. Confirm.
- **Q4 — version-assumption docstring.** The "LiteLLM 1.89.0 `_hidden_params`
  contract" note moves into the ACL module docstring (single home). Confirm the
  `enterprise_logger.py:549` `_patch_lowest_cost_none_guard` monkey-patch is
  **out of scope** for 0.5.3 (it patches a router strategy, not a read of internals).
