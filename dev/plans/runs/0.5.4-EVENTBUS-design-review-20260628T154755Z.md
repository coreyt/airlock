## Verdict: BLOCK

### 1. [high] Fathom `response_text` not derivable from the proposed event response
Refs: design-request-event-bus.md:147/157, fathom_logger.py:64/154, enterprise_logger.py:496

`_response_text()` reads `.choices` off an **object**, but the event's `response`
was described as `_serialize(response_obj)` (a top-level **dict** `{"choices": ...}`).
Running `_response_text` on the serialized dict drops `response_text` output. Fix:
carry the **raw** `response_obj` on the event (projections serialize per-sink); fathom
runs `_response_text(event.response_obj)` exactly as today. Pin with a golden test.

### 2. [medium] Fathom whole-sink skip not pinned
Refs: fathom_logger.py:229, enhanced_passthrough.py:170/172, design-request-event-bus.md:210

Fathom skips the **entire** write when `metadata["airlock_skip_fathom_logger"]` is
set (enhanced passthrough sets it for inner physical-model calls). The design must
require `emit_fathom` to preserve this skip, else unified fan-out creates extra
fathom rows. Pin with a seam/golden test (flag set → no fathom row).

### 3. [low] Registration inventory overstates s3/sql self-registration
Refs: design-request-event-bus.md:206, s3_logger.py:208, sql_logger.py:183

s3/sql expose module **instances only** — no `_self_register()`, not in config
(opt-in/inactive by default). The cutover text must distinguish self-registered
callbacks (enterprise sync+async+config; fathom **async-only**; metrics
self-register) from opt-in config sinks (s3/sql) so cutover doesn't chase
nonexistent removals — and so each sink's exact firing surface is preserved.

## What passed on inspection
s3/sql raw-failure reproduction via `bare_exception_error` correct (preserves

boundary sound; monitor-ordering claim correct (config lists enterprise before
