## Findings (pre-existing, non-blocking for 0.5.1 — recommend follow-ups)

These were surfaced by the smoke but are **not** 0.5.1 regressions — 0.5.1 changed only the
*settings source* (config-driven budgets/failover/warn-ratio), and those changes demonstrably
work live (the warn fired at the configured ratio; spend durability holds).

- **F-B [obs, low/med]:** `X-Airlock-Budget-State: near_limit` **detects + logs** correctly,
  but the **response header** did not surface on non-streaming calls. The metadata→header
  surfacing (`monitor.py:69` sets `airlock_response_headers`; `callbacks/model_override_headers.py`
  attaches them) is unchanged 0.5.0 transparency plumbing — a success-callback vs header-attach
  ordering nuance. 0.5.1 did not touch this path. The `dev/smoketest` warn-ratio scenario added
  by SET-warnratio asserts this header; that assertion needs the surfacing path confirmed.
- **F-C [routing, low]:** proactive-swap `X-Airlock-Model-Override` did not surface. The router
  swap (`router._apply_budget_awareness`) only fires when a **candidate pool** is supplied; a
  plain direct request generates none, so no swap regardless of budget. 0.5.1 changed only the
  budget/ratio source, not this gating. Pre-existing behavior.
- **F-D [config/env, low]:** `AIRLOCK_STATE_DIR` set via the harness's copied `.env` did not
  reach the runtime env, so checkpoints used the `./logs` cwd-default (`monitor.py:408` reads
  `AIRLOCK_STATE_DIR` correctly — it was just absent from the process env). Consistent for both
  checkpoint and restore, so **durability is unaffected**. Mirrors the pre-0.5.1 breaker-checkpoint
  path resolution. Worth confirming `.env`→child-env propagation for state-dir, and that a real
  exported `AIRLOCK_STATE_DIR` (not via `.env`) lands checkpoints where operators expect.

## Verdict

**0.5.1's deliverables PASS the live smoke** — settings unification boots and reads config,
spend is accurate (µ$/bucketed) and **durable across restart** (the headline FIX-1/UN-26, the
riskiest claim), the unified warn ratio fires at the configured ratio, and `0`/absent budgets
mean no enforcement (production budgets are `0`). Findings F-B/F-C/F-D are pre-existing plumbing
behaviors independent of 0.5.1's changes → tracked as follow-ups, not 0.5.1 blockers.
