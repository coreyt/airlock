### 1. [medium] Failover targets are not constrained to `model_list`
Refs: `airlock/fast/settings.py:180`, `airlock/fast/circuit_breaker.py:89,95`, `tests/test_fast_circuit_breaker.py:137`

R2 is fixed for the shipped/default path (stale `gpt-4o` defaults gone; config `fallbacks`
used). But the runtime still accepts arbitrary `AIRLOCK_FAILOVER_MAP` values, and
`check_model_with_filters()` returns the first *healthy* string from the map without
validating it exists in `model_list`. Since unknown models are healthy-by-default in the
state store, an env override or typo can fail over to a non-existent LiteLLM alias. The test
at `:137` currently locks in the arbitrary-map behavior.

## Orchestrator triage — FIX (no override)

The stated AC-R2 ("guards the stale `gpt-4o` target") IS met, but the finding is R2's deeper
spirit — a circuit-open should land on a model the proxy actually serves. Worth fixing now
since the release theme is settings coherence. **FIX**, scoped minimally:
- In the failover selection, restrict candidates to aliases known to the router's loaded
  `model_list` catalog (the same catalog `infer_provider`/`set_router_config` already cache),
  warn-and-skip any target not served — covering BOTH config-derived and `AIRLOCK_FAILOVER_MAP`
  override maps.
- **Safe fallback (required):** if the catalog is empty/unconfigured (e.g. `set_router_config`
  not called in some context), do NOT filter — preserve current behavior — so we never
  accidentally suppress all failover when the catalog simply isn't loaded. Only filter when a
  non-empty catalog is known.
- Update the `:137` test to expect filtering of unknown targets, and add a test that a
  circuit-open with a typo'd/unknown failover target skips it and lands on a real `model_name`
  (and that an empty catalog disables filtering).
