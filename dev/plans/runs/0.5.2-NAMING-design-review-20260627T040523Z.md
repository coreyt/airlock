## codex transcript — v1 (BLOCK, 3 findings)

```

The design is mostly implementable, but it is not sound as written because it misses a pre-LiteLLM Airlock alias-table collision. The LiteLLM exact-match assumption is not enough: `guardian` calls `alias_table.resolve()` and rewrites `data["model"]` before LiteLLM routing. `model_alias` also inserts bare provider-model variants into `_exact`, so multiple entries with the same stripped provider model can overwrite each other.

Specific answers:

1. Yes: `set_router_config()` maps exact `model_name` to `litellm_params.model.split("/", 1)[0]`, so `aistudio/gemini-3.5-flash` backed by `gemini/gemini-3.5-flash` maps to `gemini` without `_PROVIDER_PREFIXES`. Adding `aistudio`/`vertex` to `_PROVIDER_PREFIXES` is unnecessary and risky if mapped to display names instead of actual serving providers.

2. Not sufficient: LiteLLM may exact-match later, but Airlock’s guardrail can rewrite first. This must be fixed in `model_alias`, not only gated with a LiteLLM router test.

3. Pinning is unaffected for normal aliases: `_is_client_pinned()` treats slash aliases as concrete, and `_lock_pinned_request()` is model-string agnostic. Served-by is post-call from hidden `custom_llm_provider`, so normal AI Studio/Vertex aliases are unaffected.

4. Capability metadata is implementable, but “single source of truth” is only true if `/v1/models` and validation both use the same helper. `/model/info` is effectively config passthrough from Airlock’s perspective. Also, `batch iff vertex_ai/` is too broad unless the helper accounts for actual LiteLLM/native batch support and configured Vertex location.

5. Batch legacy/new alias dual-listing works: `load_batch_aliases()` keys by exact alias and `backend_for_alias()` resolves each exact key independently. Same backend/provider_model is not a collision.

6. The `enhanced/` special case is required. `set_router_config()` would otherwise map `gemini-coding`/`aistudio/gemini-coding` to `enhanced`, which is wrong for provider protection and Gemini semantics. The fix should be a shared `airlock_provider_for()` used by router/capability code. Served-by should still come from the inner physical Gemini call, but add a regression test.

7. The additive ASGI `/v1/models` seam is sound if installed like batch middleware and only adds an `airlock` subobject. `/model/info` should be treated as config-only unless a smoke test proves LiteLLM’s exact `model_info` passthrough for the pinned LiteLLM version.

8. Missing integration point: `model_alias` exact/variant table. This is the blocking issue.

## Verdict: BLOCK

1. [airlock/fast/model_alias.py](<repo-root>/airlock/fast/model_alias.py:253): `_exact[bare] = alias` can overwrite an explicit legacy alias or another provider alias sharing the same stripped provider model, and [airlock/fast/guardian.py](<repo-root>/airlock/fast/guardian.py:291) applies that rewrite before LiteLLM sees the request. Fix with two-pass alias loading: explicit `model_name` keys are immutable, provider-model/bare/version-stripped variants are added only when unambiguous, and ambiguous variants are not cached. Add tests proving bare legacy aliases do not repoint to `vertex/...` and native `gemini/...` inputs do not cross-provider rewrite.

2. [airlock/fast/router.py](<repo-root>/airlock/fast/router.py:305): provider inference from `litellm_params.model` misclassifies `enhanced/gemini-coding` as `enhanced`. Fix by introducing a shared Airlock provider classifier that maps `enhanced/*` Gemini profiles to `gemini`, and use it in `set_router_config()` plus capability generation. Add served-by and Gemini-semantics regression tests.

3. [config.yaml](<repo-root>/config.yaml:135): publishing `batch` for every `vertex_ai/` model can drift from actual batch support/config, especially with current `vertex_location: global` comments warning batch may require regional locations. Fix `endpoints_for()` to model actual supported routing, not just provider prefix, and validate both `/model/info` config and `/v1/models` augmentation against that helper.
tokens used
```

D looks correct: pinning is model-string agnostic, and served-by attribution is post-call from hidden LiteLLM provider metadata, not request model parsing.

## Verdict: BLOCK

[airlock/fast/model_alias.py](<repo-root>/airlock/fast/model_alias.py:310): the prefix-strip fast path can reintroduce cross-provider ambiguity for slash-prefixed misses, e.g. `vertex_ai/gemini-3.5-flash` strips to the explicit bare AI Studio alias. Fix by making prefix-strip provider-aware and ambiguity-aware: only strip when the slash prefix is compatible with the target provider or the stripped body is an unambiguous non-explicit variant; otherwise return no alias/fall through without caching.
