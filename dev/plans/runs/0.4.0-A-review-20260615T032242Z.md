## Verdict: BLOCK

### 1. [high] Caller-controlled batch markers can skip model guardrails
Refs: `airlock/guardrails/extract.py:124-128`, `airlock/fast/guardian.py:171`, `airlock/fast/guardian.py:199`

`is_batch_call()` returns True on a top-level `input_file_id` key or
`purpose == "batch"` **independent of `call_type`**. Those are caller-controlled
request-data fields, so a normal completion carrying one is classified batch and
skips alias resolution, provider protection, routing, circuit breaker/failover,
and the pinned-request lock. Tests cover positive markers but not a normal
completion carrying `input_file_id`/`purpose=batch`.

### 2. [medium] `_lock_pinned_request` skip inherits the marker-forced bypass
Refs: `airlock/fast/guardian.py:167`, `airlock/fast/guardian.py:171`

Skipping the pinned lock for *real* batch/file routes is justified, but because
`batch` can be marker-forced (finding #1), a pinned normal completion can avoid
Airlock's downstream fallback/retry lock. Resolved by fixing #1.

### 3. [low] Spec/prompt file absent in the worktree
Refs: `dev/plans/prompts/0.4.0-A-is-batch-call-seam.md`

The worktree (base `90ee9c4`) predates the `dev/plans/prompts/` tree, so the
reviewer could not read the spec. Symptom of the stale-base worktree trap, not a
code defect.
