## Verdict: CONCERN

### 1. [medium] Site 12 — a real unrecorded injection path (config-fallback)
Refs: `airlock/providers/enhanced_passthrough.py:146,155,157,171`

`EnhancedPassthroughProvider._resolve_request()` resolves an enhanced profile from the
**config cache** when `litellm_params["enhanced_profile"]` is absent
(`_resolve_profile` → `_load_profile_cache`), then injects `system_prompt` at provider
execution time (`:157`). Site 11 (`EnhancedModelInterceptor`) only records when
`litellm_params["enhanced_profile"]` is present, so this config-fallback injection can
go **entirely unrecorded** in the ledger — a real UN-19 coverage gap. No value leak
(injected text is never stored); the mutation is simply invisible. The implementer's
"duplicates Site 11" rationale does not hold for this path. Closing it cleanly needs the
**outer** request metadata at the injection point — but the only `metadata` in scope
(`:171`) is a fresh copy stamped `no_log`/`airlock_skip_fathom_logger` for the
suppressed inner `litellm.completion`; reaching the outer ledger requires either
threading a param through litellm's `CustomLLM.completion(...)` contract or teaching the
pre-call interceptor about config-resolved profiles.
