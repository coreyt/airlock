# Airlock 0.5.1 — Plan

Release theme: **settings coherence + the storage seam — one typed home and one
precedence rule for every airlock setting, behind a DualCache-backed store.** A
point release that removes the override/duplication/confusion between Airlock's
`fast/` settings and LiteLLM's native settings, **and** introduces the storage
abstraction (`DualCache`, in-memory / single-process) that the runtime state
currently lacks.

**In scope:** config coherence (R1-R4) + the **in-memory DualCache seam** in
`fast/state.py`. **Deferred (and deferred is a deliberate scope choice, not a
constraint — see "Deferred"):** only the **Redis backend** and the
**`--num_workers` passthrough**. Adopting DualCache now is cheap and
single-process-valuable — it exposes *synchronous* methods
(`get_cache`/`set_cache`/`increment_cache`, `dual_cache.py:114-184`) that drop
into the monitor's existing sync `log_success_event` path — and it makes the
later Redis + multi-worker step a config flip rather than a rewrite. (An earlier
draft conflated the DualCache *abstraction* with the Redis *backend* and cited a
sync/async "blocker" that does not hold; corrected here.)

> Motivating incident (2026-06-24): `provider_budget_config.gemini: 0` in
> `config.yaml` silenced LiteLLM's hard budget block **and** the monitor's warn,
> but the fast router kept proactively swapping Gemini away at a **hardcoded**
> `$25/day` it read from `_DEFAULT_PROVIDER_BUDGETS` — a constant that ignores
> `config.yaml` entirely. Three subsystems, one concept, three disagreeing
> sources. A stopgap (`gemini: 0.0` in `_DEFAULT_PROVIDER_BUDGETS`,
> `airlock/fast/router.py:69`) is already in place; this release removes the
> duplication that made the stopgap necessary.

Orchestrated with the agent harness — see `dev/agent-harness-runbook.md` and this
directory's `README.md`. Live board (to create): `dev/plans/runs/STATUS-0.5.1.md`.

### SET-unify: deliverables

- **Single budget source of truth:** both the router swap (`_load_provider_budgets`)
  and the monitor warn (`configure_budgets`) read `router_settings.provider_budget_config`,
  with `AIRLOCK_PROVIDER_BUDGETS` env as the only override. Delete the hardcoded
  `_DEFAULT_PROVIDER_BUDGETS`; no hidden defaults.
- **Fix R6:** correct `configure_budgets` to read the `router_settings` nesting (it
  currently reads top-level → `_configured_budgets` is always empty). Add a
  regression test asserting a budget set under `router_settings.provider_budget_config`
  is actually captured by the monitor.
- **Single failover source of truth:** derive the circuit-breaker failover map from
  `router_settings.fallbacks` (convert list-of-1-key-dicts → `dict[model→list]`);
  delete `_DEFAULT_FAILOVER_MAP` with its stale `gpt-4o` aliases (R2). Add config
  injection to `circuit_breaker.py` (mirror `set_router_config`) and wire it in
  `proxy.py`.
- **config.yaml budget-doc note (requested):** add an explanatory comment block
  above `provider_budget_config` in `config.yaml` **and** the shipped template
  (`airlock/cli/templates/config.yaml`) documenting how the budgets are used —
  specifically that one number drives **three** behaviors: (1) LiteLLM's hard
  block (request refused at limit), (2) the monitor's near-limit warn, and (3) the
  fast router's **proactive model reroute** (swap to another provider's model as
  spend approaches the limit) — and that `0` means *no enforcement / unlimited /
  no reroute* (the falsy-short-circuit contract). Note the `1d` window and the
  env override `AIRLOCK_PROVIDER_BUDGETS`.

### STORE-seam: scope guardrails (so it stays single-process and behavior-true)

- **Checkpoint/restore must run in the litellm child process, not the launcher
  (FIX-1).** Today `restore_state`/`checkpoint_state` are wired in `proxy.py`
  (`:307,:444`), which is the **launcher** process — but the store is mutated only
  inside the `litellm` **subprocess** (`proxy.py:450`) where `airlock.fast.monitor`
  runs. So the parent checkpoints an *empty* store and the child's state is never
  persisted (this already silently defeats the existing breaker checkpoint).
  STORE-seam must move/duplicate checkpoint+restore into the **runtime/callback
  process** (e.g. monitor startup + a periodic/shutdown checkpoint inside the
  child). **Acceptance: an end-to-end subprocess restart test** (start proxy →
  record spend → stop → start → assert spend restored), not just an in-process
  unit test.
- **Q3 scope is Airlock's spend, not LiteLLM's (FIX-2).** Persisting Airlock
  `ProviderSpend` does **not** restore LiteLLM's own
  `provider_spend:{provider}:{duration}` DualCache (`budget_limiter.py:430,454`),
  so LiteLLM's **hard** `provider_budget_config` enforcement would still reset on
  restart. This is moot today (all budgets `0` ⇒ LiteLLM enforcement off), so
  0.5.1 **explicitly scopes Q3 to Airlock's warn/swap spend**. If hard budgets are
  ever reintroduced, seeding LiteLLM's limiter cache from the checkpoint becomes a
  required follow-up — tracked as an open question, not silently assumed.
- **Airlock-owned cache config, not DualCache defaults (FIX-3).** A default
  in-memory `DualCache` carries a ~10-minute TTL and no locking around
  get/set/increment (`in_memory_cache.py:30,31,129`; `dual_cache.py:114,128,151`).
  For a 24h window that would **expire spend early** and **lose concurrent
  increments**. STORE-seam must set an **explicit per-key TTL ≥ the rolling
  window** on every stored spend key — **do not rely on `default_ttl=None`**, which
  LiteLLM coerces to 600s via `default_ttl or 600` (`in_memory_cache.py`) — or wrap
  the cache to truly disable TTL. Use an adequate max size, and hold the existing
  `StateStore._lock` (`state.py:685`) around compound read-modify-write updates.
- **Store spend as integer micro-dollars (FIX-4).** Sync/Redis `increment_cache`
  is integer `INCR`; costs are floats. To keep the "later Redis is a config flip"
  promise honest, accumulate spend as **integer µ$ (or cents)** rather than float
  increments — pick the unit now so the in-memory and future Redis backends agree.
- **Rolling window, not count-capped.** Replace the `deque(maxlen=1000)` with a
  time-windowed accumulator (timestamped buckets pruned by age), so `recent_spend`
  reflects a true trailing 24h regardless of call volume (fixes R5). A **tumbling**
  daily-reset key (LiteLLM's `:1d` style) is explicitly **rejected** here — it
  would zero spend at rollover, which conflicts with Q3.
- **Defined checkpoint semantics (FIX-7).** The spend checkpoint must specify:
  (a) a versioned schema; (b) **atomic write** (temp file + `os.replace`);
  (c) **prune-before-checkpoint** so stale (out-of-window) records aren't written;
  (d) **replace-not-append, idempotent restore** (rehydrating must not double-count
  if a record is already present). Restore is bounded by record age (only in-window
  records are rehydrated), **not** by the 5-minute `cb_state` freshness gate, which
  is a breaker-specific staleness guard and must remain breaker-only.
- **Keep `cb_state.json` breaker recovery working** (`state.py:1290`) once the
  process-location fix (FIX-1) lands — the seam must not break breaker restart
  recovery; spend persistence rides the same (now correctly-located) path.
- **Abstract behind our own interface**, so swapping the in-memory backend for
  Redis later (or a `ProcessPoolExecutor`-shaped future) touches one module.

### Logical, not physical, unification (the key correction)

- `provider_budget_config` **stays in `router_settings`** — LiteLLM reads it there.
  Airlock *reads it from there too*. The "single namespace" goal is **logical**
  (one loader, one precedence), not a physical key move.
- `cost_tiers` stays top-level (or dual-read) to avoid breaking existing configs.
- `airlock_settings:` gains the keys that today have **no** config home:
  `session_ttl`, `smart_thresholds`, the unified `budget_warn_ratio`, and
  (optionally) a `failover_overrides` escape hatch.

## Deferred (NOT in 0.5.1 — and these are deliberate scope choices, not constraints)

Two things wait for a later release. Both are *decisions*, not technical walls —
the STORE-seam in 0.5.1 is explicitly designed so they become incremental, not a
rewrite.

- **Redis backend for the store.** The DualCache seam ships in-memory in 0.5.1,
  with spend **persisted to disk** (Q3) via the file checkpoint — so
  restart-durability is *already solved without Redis*. Redis is therefore
  deferred to the **one** thing it uniquely enables: **shared state across
  multiple workers/instances**. Note the existing cross-process channel: the TUI
  replica already syncs via **JSONL event replay** (`state.py:1357 tail_jsonl` /
  `:1189 ingest_jsonl_record`, CC-9). A Redis store would be a *second* sync path
  for overlapping state — reconcile or supersede deliberately when it lands.
- **`--num_workers` passthrough.** Out of the box `airlock start` runs **one
  worker** — the launcher (`proxy.py:413-421`) builds the `litellm` command with
  `--config/--host/--port` (+ ssl) and no worker flag; there is no
  `AIRLOCK_NUM_WORKERS`. The underlying `litellm` CLI supports `--num_workers`
  (default 1); plumbing it is ~1 line. It is deferred because **multi-worker is
  unsafe until the store is Redis-backed** — today's per-process in-memory state
  (`state.py:685` RLock + deques) would fragment across workers and the
  `cb_state.json` restore (`proxy.py:436`) would race. So `--num_workers` and the
  Redis backend are the *same* future step, gated on each other.

### Internal-API coupling (applies whenever the Redis step lands)

`DualCache`/`RedisCache` are internal LiteLLM surfaces (`pyproject.toml:44`
declares the **range** `litellm>=1.83.4,<2`; the venv/`uv.lock` currently
**resolves 1.89.0**) and module paths churn within `<2`. The STORE-seam
abstracts them behind our own interface for exactly this reason. (`BudgetConfig` +
`duration_in_seconds` are stable enough to import directly, and do in SET-loader.)

### Window-semantics note (resolved by Q3)

The pre-0.5.1 deque was a true **sliding** 24h window but count-capped at 1000 →
undercounts (R5) **and** zeroed on restart. LiteLLM's own
`provider_spend:{provider}:1d` is a **tumbling** window that resets at rollover
(`budget_limiter.py:379-405`). STORE-seam takes neither verbatim: it keeps a
**rolling** trailing-window accumulator (fixes the undercount) **and** checkpoints
it to disk (survives restart, per Q3). The net behavior differs from both the old
deque and LiteLLM's tumbling key — document it, don't claim preservation.

## Behavior-change register (release-notes required)

All silent; each needs an explicit changelog entry:

1. Deleting `_DEFAULT_PROVIDER_BUDGETS` turns **off** proactive cost-swaps for any
   deploy that relied on the hidden `50/50/25/25` defaults (when budgets are 0/unset).
   **Resolved (Q1):** intended — budgets are config-driven; auto-swap requires an
   explicit `provider_budget_config`. Document, don't preserve hidden defaults.
2. Unifying the warn ratio moves the **router swap point** off `0.9` to the
   configured value.
3. Deriving failover from `fallbacks` changes which model a circuit-open request
   lands on (today often a non-existent `gpt-4o` → error; after → a real alias —
   a **fix**, but a routing change).
4. **Restart no longer zeroes provider spend (Q3).** Captured spend is now
   checkpointed and restored. This intentionally removes the prior "restart resets
   the budget counter" side effect — a restart will no longer clear an
   accumulated spend total.
