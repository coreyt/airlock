## Items

### F-1 — `X-Airlock-Admission` response header

**Status:** follow-up from 0.5.5 (IMPL-admission).

**Context:** The admission gate stamps `data["metadata"]["airlock_admission"]` with
`{"action": "admitted"|"shed", "retry_after": N}` on every request. This metadata
is never surfaced to the HTTP client. The intended header is `X-Airlock-Admission:
admitted` / `X-Airlock-Admission: shed; retry_after=N`.

**What needs to happen:**
- Identify where response headers are set from metadata (check `airlock/transparency.py`
  and the LiteLLM response-hook chain).
- Add `X-Airlock-Admission` header from `metadata["airlock_admission"]`.
- On shed (429 path), also set a `Retry-After` HTTP header matching the gate's
  `retry_after` value (RFC 7231 §7.1.3).
- Add test: assert header present + correct value for admitted and shed paths.

**Acceptance criteria:**
1. `curl -I` on a request that passes the gate shows `X-Airlock-Admission: admitted`.
2. A shed request returns `HTTP 429` with both `X-Airlock-Admission: shed; retry_after=N`
   and `Retry-After: N`.
3. When admission is disabled (`admission.enabled: false`), neither header is set.

### F-2 — True async semaphore acquire/release for concurrency cap

**Status:** follow-up from 0.5.5 (IMPL-admission).

**Context:** `AdmissionGate.check()` peeks at `sem._value <= 0` to detect a full
semaphore. Because `check()` is synchronous and never acquires the semaphore, the
concurrency cap is a burst detector only — it cannot enforce a hard limit on
simultaneous in-flight requests.

**What needs to happen:**
- Change the gate from a single synchronous `check()` call to an async context that
  acquires the semaphore on admit and releases it when the response completes.
- The acquire must be non-blocking (try-acquire, not `await sem.acquire()` which
  would queue). Shed immediately if no slot is available.
- Release must happen in a `finally` block tied to the LiteLLM response callback or
  an async context manager wrapping the call — needs investigation of the guardian
  hook lifecycle.
- Update `AdmissionGate` interface accordingly; update tests.

**Acceptance criteria:**
1. With `concurrency: 2` and 3 concurrent requests, exactly 1 is shed (the third).
2. After a request completes, its slot is released and subsequent requests are
   admitted again.
3. `check()` (or its replacement) still completes in < 1ms for the non-blocking path.
4. Fail-open invariant preserved: any error in acquire/release logs a warning and
   admits the request.

**Design note:** This likely requires changing the gate from synchronous to async,
which means the guardian insertion point becomes `async`. The guardian hook is
already `async`, so there is no fundamental blocker — the complexity is in ensuring
the semaphore is released exactly once per admitted request regardless of how the
call terminates (normal return, exception, timeout).

### F-3 — Helpful rejection response for unknown model names

**Status:** pulled forward from 0.5.8 P-2b. Design §4.2.2.

**Context.** 0.5.6 made `ModelAliasTable.resolve()` refuse any fuzzy match that would
ignore a qualifier the caller supplied — `gpt-5.6-mini` no longer silently routes to
Sol at 5x Luna's price. But the refusal is *unhelpful*: `resolve()` returns `None`,
`guardian.py` leaves the unknown name in `data["model"]`, and litellm emits its own
generic "model not found". The caller is not told what to send instead, despite
`ModelAliasTable.suggest()` already returning ranked candidates with cost tiers.

**Files:** `airlock/fast/guardian.py` (around the `alias_table.resolve()` call),
`airlock/proxy_errors.py`, `airlock/transparency.py`.

#### Changes

- When `resolve()` returns `None` **and** the name was refused (as opposed to simply
  unknown), short-circuit with a **404 + `code: model_not_found`**, OpenAI-shaped so
  existing SDK error handling works unchanged.
- `error.message` must be **self-sufficient prose** — most clients surface only that
  field. The structured `error.airlock.suggestions` block (model, score, tier) is a
  bonus, not the payload.
- Add `X-Airlock-Model-Suggestion` using the established `;`-joined `key=value`
  grammar. **Reuse the `_mutation_token` serializer** — do not write a second one.
- Suggestions come from `suggest()`, never a hardcoded table, so a new model family is
  suggested correctly with no code change.
- **Never suggest a model the caller is not entitled to.** Respect any per-client
  catalog filtering — a helpful error must not become a capability-disclosure channel.

#### Tests

- [ ] `gpt-5.6-mini` returns 404, not litellm's generic error.
- [ ] `error.message` alone names the rejected model, a suggestion, and the
      alternatives — readable without parsing the structured block.
- [ ] `X-Airlock-Model-Suggestion` grammar matches `_mutation_token` byte-for-byte.
- [ ] Suggestions never include a model outside the caller's permitted catalog.
- [ ] A genuinely unknown name (no near match) still returns a sensible error rather
      than an empty suggestion list.
- [ ] Names that resolve today still resolve — no regression in routing.

**User need:** UN-29 ACs 4–5.

#### Exit criteria

A caller who guesses a model name is told exactly what to send instead, in the error
body and in a header, without Airlock ever serving a model they did not ask for.

### F-4 — `enhanced/*` records $0.00 against real spend

**Status:** pulled in from `dev/notes/ticket-unpriced-models.md` (P1).

**The bug.** `enhanced/gemini-coding` really serves
`gemini/gemini-3.1-pro-preview-customtools` — **$2.00 in / $12.00 out per 1M**, with
200K threshold pricing on top. But `enhanced/gemini-coding` is not in litellm's cost
map, so `response_cost` comes back 0/None and **every one of those requests records
$0.00**. Provider budgets under-count, near-limit downgrade routing never fires for
that path, and MTD/YTD spend is wrong by the full amount. Aliases affected:
`gemini-coding`, `aistudio/gemini-coding`.

**Scope note:** fix the `enhanced/` *mechanism*, not the one model. Any future
`enhanced/*` profile inherits the same defect by construction.

#### Step 1 — establish whether there is anything to plumb (do this FIRST)

`EnhancedPassthroughHandler.acompletion` ends with
`return await litellm.acompletion(model=target_model, ...)` — it returns the **inner**
response object directly. That inner call is against a *priced* model, so its
`_hidden_params["response_cost"]` may already be correct and may already survive to
Airlock's reader at `litellm_adapter.py:60`.

**Determine empirically before writing any fix.** The two outcomes are very different:

- **Cost survives** → there is no plumbing to do. Add a regression test that pins it,
  correct the ticket, and stop. (This is plausible and would make the ticket's
  "attribution plumbing" framing wrong.)
- **Cost is lost or overwritten** → litellm recomputes for the outer custom-provider
  model and zeroes it. Proceed to step 2.

Verify with a real request if a funded key is available, or by inspecting
`_hidden_params` on the returned object with a stubbed inner call. Do **not** infer it
from the cost map alone — absence from the map explains the *outer* model having no
price, not whether the inner cost propagates.

#### Step 2 — if the cost is lost

Preference order:

1. **Propagate the inner cost.** The handler already knows `target_model` and holds the
   inner response. Re-stamp `_hidden_params["response_cost"]` (and the served-model
   attribution) from the inner call. Correct by construction and automatically tracks
   upstream price changes — no table to maintain.
2. **Register the target's pricing for the outer name** via litellm's model-cost
   registration, so the outer call prices as the target. Second choice: it duplicates a
   price that upstream already owns and will drift.
3. **Static `input_cost_per_token` / `output_cost_per_token` in `litellm_params`** —
   last resort only. It hardcodes a price that changes, and would silently go stale
   exactly like the version drift this release cycle already fixed.

Whichever lands, the recorded cost must honour the target's **threshold pricing**
(`gemini-3.1-pro-preview*` bills 2x in / 1.5x out above 200K input), not a flat rate.

#### Step 3 — related, decide explicitly

- **`tavily/web-search`** bills per *search credit*, not per token, so it will never
  appear in a token cost map. Needs a flat per-request cost. Whether Airlock's cost
  path honours litellm's `input_cost_per_request` on a custom provider is
  **unverified** — check it, then either fix or record why it is deferred.
- **The four self-hosted vLLM models legitimately cost $0** and must NOT be given a
  fake price. Consider whether "unpriced by design" should be distinguishable from
  "$0 because we failed to look it up" — today they are identical in the data, and
  only one of them is honest.

#### Tests

- [ ] A request through `gemini-coding` records a **non-zero** cost matching the
      target model's rate (or, if step 1 shows cost already survives, a regression test
      pinning that so it cannot silently break).
- [ ] Long-context (>200K input) records the **surcharged** rate, not the base rate.
- [ ] The cost lands in provider-spend accounting, not just the response object.
- [ ] Self-hosted vLLM models still record $0 — no fake pricing introduced.
- [ ] A second `enhanced/*` profile pointed at a different target prices correctly,
      proving the mechanism is fixed rather than one model special-cased.

**User need:** UN-30 ACs 1–3, 5.

#### Exit criteria

No configured model silently records $0.00 while incurring real spend, and the fix is
mechanism-level so the next `enhanced/*` profile inherits it.
