# Airlock 0.5.0 — Transparency Workstream Plan

A third leg of the 0.5.0 train, alongside resilience and the admin control plane:
**mutation & serving-backend transparency.** Make observability a first-class
*benefit* of routing through Airlock rather than a black box. Two joined gaps,
shipped together: (1) every way Airlock mutates a request is recorded in one
canonical ledger and surfaced by default (header + log + metrics); (2) Airlock
reports which backend *actually* served each request — read from the response, not
guessed from the model name — and attributes spend/quarantine to it.

> Lands on the same branch as the rest of 0.5.0 (`feat/0.5.0-resilience-admin`),
> as a follow-on workstream after the 10 resilience+admin packs (all CLOSED — see
> `STATUS-0.5.0.md`). Live board: `dev/plans/runs/STATUS-0.5.0-transparency.md`.
> Parent plan: `dev/plans/0.5.0-plan.md`.

Orchestrated with the agent harness — see `dev/agent-harness-runbook.md` and this
directory's `README.md`. The implementation session is launched from

### Per-pack scope detail

- **OBS-core** — NEW `airlock/transparency.py` (dataclasses + `record_mutation`,
  `attribute_served_backend`, `mutations_header`, `served_headers`) + config load
  for `transparency.*` (default: `mutation_headers: compact`, `served_headers: true`,
  `attribute_accounting_to_served: true`, `mutation_header_budget_bytes: 256`).
  Pure module — no wiring into the request path yet. Tests: dataclass shape,
  `record_mutation` append/ordering, `attribute_served_backend` over crafted
  `_hidden_params` for each backend (anthropic/bedrock/vertex_ai/azure/openai/
  gemini) incl. the Google AI-Studio-vs-Vertex `api_base_host` disambiguation and
  the `None`/inferred-only path, header serializer byte-bound + `…+N`, redaction
  records carry `count`/`category` and never a value, config defaults. DoD:
  importable, fully unit-tested, zero call-sites changed → behavior identical.
- **OBS-served** — MODIFY `airlock/callbacks/model_override_headers.py`. In
  `async_post_call_response_headers_hook` (fires for stream + non-stream): call
  `attribute_served_backend(response)`, stash the identity `served` into metadata,
  emit `X-Airlock-Served-By` (+ `-Region` when present). **Provider source (CC-T3):**
  `_hidden_params.get("custom_llm_provider") or getattr(response, "custom_llm_provider",
  None)` — on streams the provider is a wrapper *attribute* (`streaming_handler.py:113`),
  not yet in `_hidden_params` at header time; `api_base`/`region_name`/`model_id` are
  in the wrapper `_hidden_params`. If provider is still unknown → omit the header
  (`attribution: inferred`), don't guess. `response_cost` finalization happens in
  OBS-log's success-hook attribution, not here. Tests: unary + streaming surface the
  correct header per backend (streaming via the wrapper attribute); unknown provider →
  no header. DoD: headers default-on; no request-path change. **HITL**: operator
  confirms headers on a real Vertex-vs-AI-Studio call.
- **OBS-ledger** — MODIFY each mutation site to record via `record_mutation`/

- **OBS-headers** — MODIFY the post-call hook to serialize the ledger into

## Cross-cutting constraints (every pack)

- **CC-T1 — One ledger.** All mutation recording goes through `record_mutation`
  into `metadata["airlock_mutations"]`; no new ad-hoc keys. Existing `airlock_*`
  keys stay for back-compat.
- **CC-T2 — Default-on but safe; header-safe surfacing.** Mutation/served headers
  default-on. Redaction records are **value-free by construction** (`record_redaction`;
  the `Mutation` ctor rejects `before`/`after` when `op == "redact"`). The header
  serializer surfaces an `after` value **only** for the allowlist
  `HEADER_VALUE_FIELDS = {model, reasoning_effort, fallbacks, num_retries}`; every
  other op/field (`inject`/`rewrite`/`redact`, message/system-prompt content) renders
  as `field=<op>`/`field=<op>(<count>)` — never content. Byte-bounded with `…+N more`;
  full detail in the log.
- **CC-T3 — Served truth from the response; inference pre-call; streaming-correct
  source.** `infer_provider()` stays the basis for routing (a prediction).
  `attribute_served_backend(response)` reads `_hidden_params`, and resolves the
  provider as `_hidden_params.get("custom_llm_provider") or getattr(response,
  "custom_llm_provider", None)` (the wrapper attribute, because on streams
  `custom_llm_provider` is not in `_hidden_params` at header-flush time). Record
  carries both, with `attribution ∈ {served, inferred}`; provider unknown ⇒ header
  omitted, not guessed.
- **CC-T4 — Observe-only request path; accounting is success-vs-failure split.**
  The ledger/attribution never change whether/where a request goes. OBS-accounting
  changes only *which counter* a number lands in: on success, spend keys off
  `served`; on a failure/429 with no response, quarantine/rate-limit key off the
  error's provider, else inferred — never blocking on a missing served read.
  Attribution is to the **final served response**, not intermediate fallback
  attempts. Gated by `attribute_accounting_to_served` (default on, a documented bugfix).
- **CC-T5 — Body compatibility.** Response **body** is unchanged by default; the
  `airlock.mutations` envelope appears only on `X-Airlock-Explain: 1` and is
  **non-streaming only** (an SSE stream's headers are already flushed and its chunk
  schema can't carry a trailing envelope).
- **CC-T6 — Streaming first-class (split surface).** Served identity + pre/during-call
  mutation headers ride the response-headers hook (flushed before the streamed body,
  provider read from the wrapper attribute per CC-T3); `response_cost` + post-call
  mutations + the full ledger/served log record ride `async_log_success_event` on
  the assembled stream. The logger attributes **independently** (it calls
  `attribute_served_backend` itself; no cross-callback ordering dependency).
- **CC-T7 — Backward-compatible, additively.** `transparency.*` absent → the
  response **body** and existing behavior are unchanged; the only wire change is the
  additive default-on headers + log fields, each with a one-line opt-out.
