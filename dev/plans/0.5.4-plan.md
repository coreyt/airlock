# Airlock 0.5.4 — Plan

> **Split out of 0.5.3 at the 0.5.3 kickoff (2026-06-27).** The observability
> event-bus unification was carried in 0.5.3 as a *candidate/stretch* item (audit
> Tier 3 #8); at kickoff it was deferred to its own point release because it is
> independent and sizable. It becomes **0.5.4**, and the former 0.5.4
> (bulkhead/isolation) was renumbered to **0.5.5**.

Release theme: **observability event-bus unification — one `RequestEvent` model and
one recorder/dispatcher behind every telemetry sink.** Airlock today rebuilds the
same per-request record independently in **four** logger sinks (enterprise / fathom
/ s3 / sql), each with its own `_build_record()`, and tracks mutations and metrics
counters through still other paths. The fields agree only by convention, so a new
field or a fix has to be applied in four-plus places and they drift. This release
collapses that duplication behind a **single canonical event + a single dispatch
seam** so every sink consumes one shared event instead of re-deriving the record.

> Motivating audit finding (2026-06,
> `dev/notes/architecture-audit-0.5.0-2026-06.md` Part 2, **Telemetry /
> event-driven ★★ — "Weakest"** row, and **Tier 3 #8**): "Unify observability
> behind one `RequestEvent` + recorder; collapse the 4 `_build_record()`s." The
> four builders + the separate mutation ledger and metrics counters are the
> concrete debt; the fix is structural, not a format change.

**This is a behavior-preserving structural refactor.** Each sink must still emit the
**same fields it emits today** — the wire/log/metrics output is unchanged. The point
is to make the record *built once* and *fanned out*, not to change what any consumer
sees. Any field that does change shape (added/renamed/dropped) is a client-visible /
logged-surface change and **must** be reflected in the smoke harness
(`dev/smoketest/`) and the behavior-change register — but the goal is zero such
changes.

**In scope:**
- A canonical **`RequestEvent`** model that carries every field the current four
  `_build_record()` implementations + the mutation ledger + the metrics counters
  derive, sourced once per request.
- A **single recorder/dispatcher** seam that builds the `RequestEvent` once and
  fans it out to all registered sinks (enterprise, fathom, s3, sql) plus the
  mutation ledger and metrics.
- Migrating the four sinks to **consume the shared event** and deleting their
  private `_build_record()` derivations.
- Golden/equivalence tests proving each sink's emitted record is byte-for-byte (or
  field-for-field) identical before vs after.

**Deferred (deliberate):** any new telemetry field, sink, or format; a pub/sub or
async event bus beyond what one in-process dispatcher needs; sink configuration
redesign. This release only *unifies the seam* — new capabilities ride later.

Orchestrated with the agent harness — see `dev/agent-harness-runbook.md`,
`dev/agent-harness-reference.md`, and this directory's `README.md`. Parent train:
the 0.5.x line (stacks on the train; confirm branch at kickoff). The orchestrator
prompt and STATUS board for this release are authored **at its kickoff**, not now.

## Why one event + one recorder

The four `_build_record()` builders are the audit's clearest "multiple disagreeing
sources" smell on the telemetry side: the same request is described four-plus times,
so a field is correct only if every copy is kept in sync by hand. Collapsing them
behind one `RequestEvent` makes the record's shape a single typed contract, makes a
new field a one-line addition, and makes every sink a pure consumer (build once,
dispatch many). It also gives later work (e.g. a real async event bus, new sinks, or
the TUI replay path) one seam to extend instead of four to chase.

The **central design decision to settle before any TDD** is the canonical
`RequestEvent` shape (which fields, sourced from where, at what point in the request
lifecycle) and the **single dispatch seam** (where the event is built, how sinks
register, ordering/failure-isolation semantics so one failing sink can't break the
request or the others). Pin that in a short design note first; everything else is
mechanical migration against golden tests.

### Testing harness (client-facing / logged surface)

The intended outcome is **no** change to any served or logged field — so the smoke
harness primarily acts as a **parity oracle** (same as 0.5.3): an isolated-instance
run before/after confirms the logged surface is unchanged. **If** the unification
unavoidably changes a logged/served field, the responsible `MIGRATE-*` pack
**must** extend `dev/smoketest/` to assert the new shape and register it in the
behavior-change register; the client keeps its production-safety guards (refuses
`:4000`/`:8090`, copied config, separate port — [[airlock-production-safety]]).

### Implementation guardrails

- **Behavior-preserving is the contract.** Each sink's emitted record is proven
  field-for-field identical before vs after by a golden test. A diff in any sink's
  output that is not an intentional, registered change is a review **BLOCK**.
- **Build once, dispatch many.** No sink re-derives fields the event already
  carries; deleting the four `_build_record()`s is the deliverable, not optional.
- **Sink failure isolation.** One sink raising must not break the request path or
  the other sinks (mirror the perimeter's "never raise from telemetry" posture).
- **No new parallel state.** The event is sourced once from existing request
  state — do not introduce a second source of truth for any field.
- **Hot-path cost.** Building one event + dispatching must not add measurable
  latency vs the current four builders; if anything it should be cheaper.

## Test surface

- **EVENT:** the dispatcher seam — sink registration, dispatch order, per-sink
  failure isolation (a raising sink is contained).
- **MIGRATE-*:** per-sink golden equivalence (enterprise/fathom/s3/sql + mutation
  ledger + metrics) — same fields, same values, before vs after.
- **VERIFY:** cross-sink equivalence over a representative request set; isolated
  `dev/smoketest/` parity run; smoke extended only if a field changed.

## Definition of Done (production-ready)

0.5.4 is **production-ready** when, recorded on `STATUS-0.5.4.md`:

1. **Design closed** — `RequestEvent` shape + dispatch seam settled in the design
   note with a codex PASS.
2. **All packs CLOSED** (EVENT, the MIGRATE-* set, VERIFY, DOCS) with a promoted
   **codex PASS** each.
3. **Acceptance criteria met** — UN-28, AC-EQUIV, AC-SEAM, AC-SMOKE all green.
4. **Equivalence proven** — every sink's emitted record is field-for-field
   identical before vs after (or any delta is intentional, registered, and
   smoke-covered); the four `_build_record()`s are gone.
5. **Full suite green, no-network**; sink-failure isolation test-covered.
6. **Isolated-instance parity run green** via `dev/smoketest/` on a separate
   dir+port — live `:4000` never touched ([[airlock-production-safety]]).
7. **Docs shipped** — UN-28 in `dev/user-needs.md`; the as-built
   `design-request-event-bus.md`; changelog + behavior-change register.
8. **Nothing pushed/tagged** without human approval; STATUS sign-off recorded.
