# Airlock 0.5.4 — Plan

Release theme: **bulkhead / isolation — stop a noisy or heavy client from
degrading everyone else, and decide *how* before building it.** Airlock today has
strong *fault* isolation (per-client circuit breaker, bounded fallback — UN-14/15/18)
but **no *resource/throughput* isolation**: a single subprocess, one event loop, no
admission control, no fair share. This release is deliberately **two-phase**: a
funded **exploration + trade-off study** that picks the right mechanism, then the
**implementation** of the chosen approach. We do not pre-commit to a design — the
whole point is that the right answer (process-level? in-loop admission control via
`PrioritySignal`? `DualCache`-coordinated? multi-process orchestration?) is not yet
known and the alternatives have real, different costs.

> Motivating audit finding (2026-06-25,
> `dev/notes/architecture-audit-0.5.0-2026-06.md` Part 2, Bulkhead row): resource
> isolation is absent; the single concrete proposal — *"Per-client RPM/TPM
> pre-emptive throttling (vs reactive quarantine)"* — is parked out-of-scope in
> `dev/notes/design-circuit-breaker-per-client.md:189`. `fast/priority.py` already
> **computes** a `PrioritySignal` but nothing **enforces** it: `priority.py:18`
> notes "downstream routing or queue-priority logic can act on it" — that consumer
> does not exist.

**In scope:** (Phase 0) the exploration/experimentation study + a reviewed design
decision; (Phase E) the implementation of whatever Phase 0 selects, plus the
**enabling refactors** that the chosen approach requires (most candidates need the
`StateStore` de-globalized — audit Tier 3 #7 — and likely the `state.py` god-object
split — Tier 3 #9; both are folded here because they are isolation prerequisites,
not independent goals).

**Deferred (deliberate):** anything Phase 0 explicitly rejects; a full Redis
deployment story beyond what the chosen mechanism strictly needs (the 0.5.1
STORE-seam already abstracts the store so Redis is a config flip when required).

Orchestrated with the agent harness — see `dev/agent-harness-runbook.md`,
`dev/agent-harness-reference.md`, and this directory's `README.md`. Parent train:
the 0.5.x line (stacks on the train; confirm branch at kickoff). Live board (to
create): `dev/plans/runs/STATUS-0.5.4.md`.

## Why this needs exploration before engineering

The candidate mechanisms differ on **isolation strength, latency cost, blast
radius, and operational complexity** — and they interact with decisions already
made (single in-memory `StateStore`, JSONL event replay for the TUI, 0.5.1's
deferral of `--num_workers`). Picking wrong is expensive: process isolation buys
true CPU/GIL isolation but fragments state across workers (the exact reason 0.5.1
deferred `--num_workers`); in-loop admission control is cheap to ship but does
nothing for a CPU-bound stall (e.g. the Presidio block 0.5.3 fixes). So Phase 0 is
funded, time-boxed, and produces **evidence**, not opinion.

## Candidate approaches (the Phase-0 agenda — to be evaluated, not assumed)

| # | Approach | Isolates… | Strength | Cost / risk | Interacts with |
|---|---|---|---|---|---|
| C1 | **In-loop admission control** — per-client concurrency gate (`asyncio.Semaphore`) + token-bucket RPM/TPM in the guardian pre-call, **consuming `PrioritySignal`** | concurrent in-flight + request rate per client | cheap, ships in-process, no deployment change | does **not** isolate CPU/GIL stalls; fairness only as good as the loop's own fairness; needs careful 429-vs-queue semantics | `fast/priority.py`, `fast/guardian.py`, the breaker (must not double-count) |
| C2 | **Process-level isolation** — `--num_workers` passthrough (0.5.1-deferred) + a shared store | CPU/GIL across workers; OS-scheduled fairness | true parallelism; real bulkhead | **fragments `StateStore`** → needs Redis/shared store; `cb_state.json` restore races (0.5.1 §Deferred); heaviest ops cost | 0.5.1 STORE-seam + Redis; `proxy.py` launcher; checkpoint/restore (FIX-1) |
| C3 | **`DualCache`-coordinated counters** — does the 0.5.1 store seam already give us cross-worker rate/concurrency accounting cheaply? | shared rate/quota state across workers | reuses 0.5.1 investment; makes C2 viable | DualCache is *accounting*, not *enforcement* — still needs C1's gate or C2's processes to act on it; sync/async surface caveats | 0.5.1 STORE-seam; C1/C2 as the actuator |
| C4 | **Multi-process orchestration** — a supervisor/worker pool with per-client or per-tier worker affinity (bulkhead pools) | full per-tier resource pools | strongest isolation; per-tenant SLAs possible | most complex; routing/affinity layer; overkill unless multi-tenant SLAs are a real requirement | C2 + a router/affinity layer; ops/deployment |
| C5 | **Hybrid** — C1 in-loop gate for the common case + C2/C3 for CPU/parallelism when scale demands | both axes, staged | pragmatic; matches "config flip" philosophy | two mechanisms to reason about | all of the above |

Phase 0 must answer, with evidence: **which axis actually hurts us first** (event-loop
contention vs CPU/GIL vs cross-worker state), **at what load**, and **which
mechanism's cost is justified by Airlock's real deployment shape** (single-tenant
dev proxy today; is multi-tenant/horizontal-scale actually coming? — the 0.5.1 open
question #2 is still open and this release should close it).

Executed by an **orchestrator** (`dev/plans/prompts/0.5.4-ORCHESTRATOR.md`) that
coordinates and never implements. 0.5.4 is **two-stage** and the orchestrator must
respect the decision gate between them — it does **not** author any Phase-E pack
prompt until the human has chosen a mechanism:

- **Phase A — Requirements + acceptance criteria.** UN-23/UN-24 below land in
  `dev/user-needs.md` at `DECIDE` (confirm next-free numbers).
- **Phase 0 (= Phase B/C/D fused) — Exploration → Design → Review → Decision.**
  `EXPLORE` runs the time-boxed study (throwaway spikes, separate dir+port) and
  produces `dev/notes/design-bulkhead-isolation.md` (the decision memo). **codex
  design-review PASS required** on the memo, then **`DECIDE` is a HITL gate**: the
  human picks the mechanism (C1–C5) and answers the horizontal-scale question. This
  is the central decision of the release; everything downstream is provisional
  until it closes.
- **Phase E — Per-pack TDD** (authored only after `DECIDE`): `ENABLE-stateprovider`
  → `ENABLE-statesplit` → the mechanism-specific `IMPL-*` packs → `DOCS`. Standard
  RED → GREEN → LINT → COMMIT → CLOSE + codex review PASS → merge per pack.
- **Release sign-off — production-ready.** See "Definition of Done" below.

Live board: `dev/plans/runs/STATUS-0.5.4.md` (one docs-commit per transition).

### Testing harness (client-facing surface)

0.5.4 ships **genuinely new** client-facing surface (mechanism-dependent, but for
the likely C1/C5: an **`X-Airlock-Admission`** response header and a **`429 +
Retry-After` shed** when a client is throttled). New client-facing surface **must**
be covered by the smoke harness, not only unit tests:

- **Extend `dev/smoketest/`** (`served_header_client.py` + `run_isolated_instance.sh`)
  with a **noisy-neighbor scenario**: drive concurrent load from a "noisy" client
  while a "victim" client measures p99, and assert (a) the victim's latency stays
  protected, (b) the noisy client receives `429 + Retry-After` (never a hang), (c)
  the `X-Airlock-Admission` header reports the decision, and (d) the gate **fails
  open** when the limiter is disabled/errors. This *is* the UN-23 load-test surface.
- The harness extension is authored **by the IMPL-\* pack** that introduces the
  surface and is finalized in **`DOCS`**; the client must keep its production-safety
  guards (refuses `:4000`/`:8090`, copied config, `/health/liveliness`).
- If `DECIDE` selects a non-admission mechanism (e.g. C2 multi-worker with no new
  header), the harness extension is **re-specified at `DECIDE`** to assert whatever
  client-visible behavior that mechanism does change (or, if none, the harness
  serves as the parity oracle as in 0.5.3). Either way, **release sign-off requires
  a green isolated-instance run covering the chosen surface.**

### 0.5.4-EXPLORE — scope guardrails (so the study produces evidence, not vibes)

- **Time-boxed.** Fixed budget; the deliverable is a decision memo with numbers,
  not a finished mechanism. Spikes are throwaway (separate dir/port,
  [[airlock-production-safety]]) and are **not** merged to the train.
- **Measure the real axes.** At minimum: (a) p50/p99 latency of N concurrent
  requests through the *current* guard chain (post-0.5.3, so Presidio is already
  offloaded — measure the residual contention); (b) CPU/GIL behavior under
  semantic-guard / PII load; (c) the cost and correctness of cross-worker counters
  via the 0.5.1 `DualCache` seam. Record the load at which each axis becomes the
  bottleneck.
- **Answer the deployment question.** The right mechanism depends on whether
  multi-tenant / horizontal scale is real (0.5.1 open-Q #2). EXPLORE must surface a
  recommendation *and* the human must answer it in DECIDE — do not build C2/C4 on a
  hypothetical.
- **Each candidate gets an explicit kept/rejected verdict with rationale** in the
  memo. A rejected approach with a one-line reason is a successful output.

### Implementation guardrails (apply once DECIDE picks the mechanism)

- **Isolation must not become a new self-inflicted outage.** Any admission gate
  fails *open* on internal error (never block a request because the limiter
  crashed), mirrors the breaker's "never raise `RateLimitError` from the perimeter"
  rule, and emits a clear `429 + Retry-After` (not a silent hang) when it sheds
  load. PII redaction and the breaker remain non-bypassable.
- **Consume, don't duplicate, `PrioritySignal`.** If C1 is chosen, the gate reads
  the existing signal (`fast/priority.py`) rather than recomputing client state —
  and must not double-count against the breaker's rate-limit accounting.
- **Honor the existing state model.** The gate's counters live behind the 0.5.1
  store seam (so single-process today, Redis-ready later) — no new parallel state
  store (that would recreate the audit's "multiple disagreeing sources" smell).
- **Latency budget.** The admission check is O(1) per request and adds <1ms on the
  hot path; prove it in the load test. The whole release is about *protecting*
  latency under load — it must not *add* baseline latency.

All experiments and load tests run on a **separate dir + port with a copied
config** — never the live `:4000` service ([[airlock-production-safety]]). The
load/concurrency harness extends `dev/smoketest/`.

### HITL gates (halt and ask the human)

- **After 0.5.4-EXPLORE** — the human chooses the approach from the decision memo

  noisy client no longer degrades a victim client's p99, and that the gate sheds
  with `429 + Retry-After` rather than hanging.
- **Release sign-off** — suite green; isolation guarantees documented (including
  what is explicitly *not* isolated).

## Definition of Done (production-ready)

0.5.4 is **production-ready** when, recorded on `STATUS-0.5.4.md`:

1. **Phase 0 closed** — decision memo codex-PASSed, mechanism chosen by the human,
   horizontal-scale question answered, UN-23/UN-24 criteria finalized to match the
   chosen mechanism.
2. **All Phase-E packs CLOSED** (ENABLE-stateprovider, ENABLE-statesplit, the
   IMPL-* set, DOCS) with a promoted **codex PASS** each.
3. **Acceptance criteria met** — AC-DECISION, AC-ENABLE, and the (mechanism-final)
   UN-23/UN-24 all green.
4. **Noisy-neighbor load test passed via the extended `dev/smoketest/`** on a
   separate dir+port: victim p99 protected; gate fails open; sheds with `429 +
   Retry-After`; the `X-Airlock-Admission` header asserted; <1ms baseline add —
   live `:4000` never touched ([[airlock-production-safety]]). The harness extension
   is itself part of the deliverable.
5. **Full suite green, no-network**; enablers proven behavior-preserving.
6. **Docs shipped** — UN-23/UN-24 in `dev/user-needs.md`; the as-built
   `design-bulkhead-isolation.md`; an ops guide stating **what is and is not
   isolated**; changelog + behavior-change register.
7. **No new self-inflicted-outage surface** — the implementation guardrails
   (fail-open, shed-not-hang, no parallel store) are each covered by a test.
8. **Nothing pushed/tagged** without human approval; STATUS sign-off recorded.

## Open questions for the human (Phase-0 agenda)

1. **Is horizontal scaling / multi-tenant SLA actually a requirement?** (Carries
   over from 0.5.1 open-Q #2 — this release must close it; it decides C1 vs C2/C4.)
2. **Fairness policy:** equal-share across clients, tiered (priority classes via
   `PrioritySignal`), or per-client configured quotas?
3. **Shed vs queue:** under contention, prefer fast `429 + Retry-After` (simpler,
   honest backpressure) or bounded in-proxy queueing (lower client-visible error
   rate, more complexity)?
4. **Acceptance:** propose **UN-23 (predictable latency under load / noisy-neighbor
   protection)** and **UN-24 (per-client resource fairness)** in `dev/user-needs.md`
   at DECIDE — confirm wording.
