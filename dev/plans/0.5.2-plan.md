# Airlock 0.5.2 — Plan

Release theme: **provider-explicit model naming + machine-discoverable
capabilities — make "which provider, and can it batch?" answerable from data,
not from a confusing alias suffix.** A point release that adopts a
`provider/model` **prefix** convention **across the entire catalog** (replacing
the `-aistudio` / `-vertex` / `-batch` **suffix** conventions), and publishes
per-model capability metadata so a client can **discover → pin → verify** without
guessing from the model string.

**In scope:** (1) provider-prefixed `model_name` aliases for **every** model
(anthropic/openai/aistudio/vertex/mistral/perplexity/tavily/vllm) with a
documented back-compat window; (2) per-model `model_info` capability blocks
surfaced on LiteLLM's `/model/info`; (3) airlock augmentation of `GET /v1/models`
to carry the same capability fields; (4) the documentation set (requirements,
project design note, user-facing guides + header catalog + a discover→pin→verify
recipe). The full per-alias migration is enumerated in **Appendix A**.

**Deferred (deliberate scope choices, not constraints — see "Deferred"):**
**removal** of the legacy bare/suffix aliases (0.5.2 only *deprecates* them; a
later release deletes them after the window); and any **client-SDK helper** for
capability discovery.

> Motivating thread (2026-06-24): a researcher pinned `gemini-3.5-flash-aistudio`
> for a sweep, saw it hang while plain `gemini-3.5-flash` worked, and concluded it
> was "a separate broken deployment." It is not — on the sync path both resolve to
> `gemini/gemini-3.5-flash` with the same key (`config.yaml:77,150`); the batch
> gateway only intercepts `/v1/batches`+`/v1/files` on `?custom_llm_provider=`
> (`airlock/batch/middleware.py:41-108`). The suffix conflates **two orthogonal
> axes** — *provider/quota* (AI Studio vs Vertex) and *API surface* (sync vs
> batch) — and encodes neither discoverably. The fix: prefix = provider, and ship
> capabilities as data.

Orchestrated with the agent harness — see `dev/agent-harness-runbook.md`,
`dev/agent-harness-reference.md`, and this directory's `README.md`. Parent train:
the 0.5.x line (stacks on `feat/0.5.0-resilience-admin`, same branch unless the
orchestrator cuts a fresh one at kickoff). Live board (to create):

### NAME-aliases — scope guardrails

- **Whole catalog, provider-by-provider.** Every entry in `model_list` gains a
  `provider/model` alias per Appendix A. Implement and test one provider family at
  a time (anthropic → openai → gemini → mistral → perplexity → tavily → vllm)
  inside the pack so a failure is isolated; the smart-router/`enhanced` aliases are
  handled explicitly (they are not bare provider models).
- **Dual-list, don't rename in place.** Add new `provider/model` entries; keep the
  old bare/`-aistudio`/`-vertex`/`-batch` entries pointing at the same
  `litellm_params` (+ same `airlock_batch` marker) for one deprecation window.
  LiteLLM permits multiple `model_name`s → same params. Mark the old ones
  deprecated in `model_info`.
- **Pick prefixes that do not collide with native LiteLLM provider tokens.** Use
  `aistudio/` (native token is `gemini/`) and `vertex/` (native token is
  `vertex_ai/`) **specifically because they differ** — so the router exact-matches
  the alias and never re-parses it as provider routing. **This is a load-bearing
  assumption: prove it with a test** on a separate dir+port before relying on it.
- **Choose and document the bare default.** `gemini-3.5-flash` (no prefix) =
  default provider (decide which; AI Studio is the incumbent). Document it as
  ops-repointable; suffixed/prefixed names are the stable client contract.
- **Migrate every internal reference atomically:** `router_settings.fallbacks`
  and `cost_tiers` entries that name the old aliases. A stale reference = a
  fallback/tier target the proxy doesn't serve (the R2-class bug from 0.5.1).

### CAP-modelinfo — single source of truth (the anti-duplication rule)

- `endpoints` must be **derived from or validated against** the real wiring, not
  hand-typed independently: `batch ∈ endpoints` iff the entry carries an
  `airlock_batch` marker (gateway path) **or** is a vertex-batch-capable
  `vertex_ai/` model. A config-consistency test asserts the published capability
  matches the actual routing — so we don't recreate the 0.5.1 "three disagreeing
  sources" problem in a new place.
- Capability schema (design finalizes): `provider` (the `infer_provider`/served-by
  value), `region?`, `endpoints: [chat|batch|…]`, `underlying` (ties the bare +
  prefixed + deprecated aliases of one model together).

### CAP-v1models — additive only

- The augmentation is **purely additive** (`object: "model"` + an `airlock` sub-
  object); it must not remove or rename standard fields. A client ignoring the
  `airlock` key sees an unchanged OpenAI `/v1/models`.
- Reuse the dual pre-start/post-start install discipline already proven for the
  batch gateway (`batch/middleware.py:546-583`) so import-order can't break it.

## Test surface (all no-network)

- **Slash-alias resolution:** a request to `aistudio/gemini-3.5-flash` and
  `vertex/gemini-3.5-flash` exact-matches the alias and is **not** re-parsed as
  native provider routing; old `-aistudio`/`-vertex` names still resolve (N5).
- **Pin preservation:** a prefixed concrete name is auto-pinned (fallbacks/retries
  off; 429 not swap) exactly as a bare name (`guardian.py:82-90,193-203`).
- **Provider inference / attribution:** `infer_provider(alias)` and the served-by
  attribution agree on `gemini`/`vertex_ai` for both old and new aliases (N2).
- **Capability ↔ wiring consistency:** published `endpoints` for each entry match
  the real routing (`batch` iff `airlock_batch` marker or vertex-batch) — the
  single-source-of-truth assertion (N3).
- **`/model/info` + `/v1/models`:** both expose the capability object; the
  `/v1/models` augmentation is additive (standard fields intact) (N4).
- **Reference integrity:** every `fallbacks`/`cost_tiers` target is a real
  `model_name` after migration (guards the stale-target class of bug).
- **Batch create:** `POST /v1/batches?custom_llm_provider=aistudio` with the new
  alias and the old alias both resolve the same backend (`gateway.backend_for_alias`).
