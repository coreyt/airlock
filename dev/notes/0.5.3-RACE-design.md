# Pack 0.5.3-RACE — design memo

## 1. threat_score atomic RMW (PRIORITY)

### Problem
`threat_detector.assess_threat` did the read-modify-write of `client.threat_score`
(and `client.backoff_until`) **outside any lock** (old lines 112–133):

    last = list(client.request_times)
    elapsed = now - last[-2]  # or 1.0
    decay = DECAY_FACTOR ** max(elapsed, 0.01)
    combined = max(score, client.threat_score * decay)
    client.threat_score = combined
    ... if blocked: client.backoff_until = now + backoff

Concurrent same-client requests can interleave the read of `threat_score` and the
write, losing an update.

### Fix — store mutator
New `StateStore.record_threat_score(client, score, now, *, decay_base, threshold,
base_backoff_s, max_backoff_s) -> tuple[float, bool, float]`:

- Runs the whole RMW under `self._lock` (the existing `StateStore._lock` RLock).
- Snapshots `client.request_times` under the lock, computes `elapsed`, `decay`,
  `combined = max(score, prev * decay)`, writes `threat_score`, computes
  `blocked`/`backoff_seconds`, writes `backoff_until` when blocked.
- Returns `(combined, blocked, backoff_seconds)`.

`assess_threat` keeps the four heuristics (deque scans / payload / error-rate) and
the `logger.warning` **outside** the lock; it computes `score`/`reasons`, then calls
`store.record_threat_score(...)` for the atomic RMW. Constants are passed in as
kwargs so `state.py` need not import `threat_detector` (avoids an import cycle).

Behavior preserved: single-threaded `threat_score`/`blocked`/`backoff_seconds` are
byte-identical (same formula, same constants, same elapsed/decay derivation).

## 2. Tests
- `tests/test_threat_score_race.py`
  - `test_record_threat_score_single_outer_lock_scope` — `_CountingLock` proves the
    RMW body is wrapped in exactly one outer lock acquisition (atomic).
  - `test_no_lost_update_concurrent` — 1 high-score thread + N low-score threads
    hammer the same client; with decay neutralised (request_times preset so
    elapsed→0.01, decay≈0.99977) the final score is deterministically ≈0.9. Without
    the lock a lost high write would drop it toward 0.1. RED: `record_threat_score`
    does not exist (AttributeError).
- `tests/test_state_stress.py` — `test_record_threat_score_concurrent_no_lost_update`
  (stress-marked) mirrors the above under the explicit stress marker.
- `tests/test_fast_threat_detector.py` —
  `test_single_threaded_characterization_unchanged`: pins `threat_score≈0.7816`,
  `blocked=True`, `backoff=256.0` for `threat_score=0.8`, empty request_times, no
  text. Stays green through the refactor.

## 3. Client-identity consolidation
Three legacy paths: `fast/state.normalize_client_id`, the inline guardian
extraction (`_request_client_id` + `_extract_client_id`), and
`client_identity.extract_airlock_client_from_headers`; `enterprise_logger`
(READ-ONLY) already routes through `client_identity` + `state.normalize_client_id`.

Canonical home = `client_identity.py`:
- `NO_CLIENT_ID`, `normalize_client_id` move here (canonical).
- `client_id_from_api_key(user_api_key_dict)` (the `key:<last8>` fallback).
- `extract_airlock_client_from_request(data, user_api_key_dict)` (the full
  request-level resolution: metadata.airlock_client → data headers → metadata
  headers → api-key fallback → no_client).

Routing:
- `fast/state.py` re-exports `NO_CLIENT_ID` + `normalize_client_id` from
  `client_identity` (so the many `from airlock.fast.state import ...` callers and
  `state.NO_CLIENT_ID` keep working; no cycle — client_identity imports no airlock).
- `guardian._extract_client_id` / `_request_client_id` become thin delegators to the
  canonical functions (kept by name because `test_fast_guardian` imports them, and
  `monitor.py`/guardrails copies are DO NOT TOUCH).

Golden test (`tests/test_client_identity.py`): parametrized inputs (header/metadata
variants incl. missing/empty/whitespace, x-airlock-client, api-key fallback) assert
`extract_airlock_client_from_request` + `normalize_client_id` equal the captured
legacy outputs, and that `enterprise_logger._get_airlock_client` agrees (parity
oracle for the 3rd path). Characterization oracle (cannot fail pre-change since the
canonical fns are new, but locks behavior against future drift).

## 4. Config-loading consolidation
