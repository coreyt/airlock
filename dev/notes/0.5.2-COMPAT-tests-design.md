# 0.5.2-COMPAT-tests — design memo

Tests-only, cross-cutting regression suite locking the whole 0.5.2 naming +
capability contract against the **real** `config.yaml` and the shipped helpers.
ZERO product/config changes. NO-NETWORK: load config, call pure helpers, drive
the `/v1/models` ASGI seam in-process via `httpx.ASGITransport`. Never start the
proxy, never hit a provider.

## File
`tests/test_0_5_2_capability_compat.py`

## Import surface (all already shipped on this branch)
- `airlock.fast.model_alias.alias_table` — `.load_from_config(path)`, `.resolve(name)`
- `airlock.fast.router.set_router_config(cfg)` + `infer_provider(name)`
- `airlock.capability.capability_record/endpoints_for/airlock_provider_for`
- `airlock.batch.gateway.load_batch_aliases(cfg)`
- `airlock.proxy._prepare_runtime_config(path)` — returns `(runtime_path, temp_path|None)`,
  injects `model_info` = `capability_record(entry)` into a temp YAML copy.
- `airlock.models_seam._build_capability_map()` / `ModelsCapabilityMiddleware`

Session fixture loads `config.yaml` once and calls `set_router_config(cfg)` +
`alias_table.load_from_config("config.yaml")`.

## The 7 locked areas
1. **Old↔new alias parity** — `infer_provider` agrees for legacy alias and the new
   `provider/model` alias (gemini/vertex_ai/mistral/openai/anthropic families);
   every legacy suffix-twin (`-aistudio`/`-vertex`/`-batch`) still `resolve()`s
   non-None.
2. **Collision-safety on the real catalog** — bare `gemini-3.5-flash` → AI-Studio
   underlying (`gemini/…`); `vertex/…` and `vertex_ai/…` → a `vertex_ai/…` entry;
   `aistudio/…` and `gemini/…` → a `gemini/…` entry; ambiguous body
   `gemini-3.1-pro-preview` (both providers, no explicit alias) → `None`.
3. **Batch backend parity** — `load_batch_aliases(cfg)` carries BOTH the new
   consolidated keys and the legacy twins with matching `backend`/`provider_model`
   for all three gateway families; bare sync entries are absent (no marker).
4. **Capability↔wiring consistency for EVERY entry** — `"batch" in endpoints_for`
   iff (truthy `airlock_batch` OR `vertex_ai/` model with a regional
   `vertex_location`). Vertex-at-`global` entries are `["chat"]`; every
   `airlock_batch`-marked entry is `["chat","batch"]`.
5. **Two capability surfaces agree (single source)** — the `model_info`
   `_prepare_runtime_config` injects and the `airlock` record
   `_build_capability_map()` produces are BOTH `== capability_record(entry)` for
   representative entries.
6. **`/v1/models` seam is additive** — drive `ModelsCapabilityMiddleware` over a
   canned OpenAI list response (ids from the real catalog) via ASGITransport;
   each model keeps `id`/`object` and gains a correct `airlock` object; a
   non-`/v1/models` path passes through unchanged.
7. **Reference integrity** — every `router_settings.fallbacks` target and every
   `cost_tiers` member is a live `model_name` in `model_list`.

## TDD / RED evidence (regression-lock pack)
The product is already implemented, so the suite is GREEN on first write — that
is the expected outcome for a regression-lock pack (no separate RED commit). The
load-bearing assertions were proven able to go RED by temporarily inverting the
expected value locally and confirming failure, then restoring:
- **Collision-safety:** asserting `resolve("gemini-3.1-pro-preview") == "x"`
  (instead of `None`) fails — confirmed `None` is enforced.
- **Old↔new parity:** asserting `infer_provider("vertex/gemini-3.5-flash") ==
  "gemini"` fails — confirmed the real value is `"vertex_ai"`.
- **Capability↔wiring:** flipping the `expect_batch` rule (forcing every entry to
  expect batch) fails on the bare sync entries — confirmed `endpoints_for`
  matches the rule exactly (0 mismatches over all 73 entries when correct).
All inversions were reverted before commit; the committed suite is green.

## Scope
No product/config edits. If any assertion failed against shipped code it would be
a real `[DETECT]` → STOP/escalate. None did.
