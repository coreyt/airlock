# 0.5.1-SET-warnratio — design memo

## Goal (R3)
Collapse the two divergent "near budget" warn ratios into ONE configurable
`budget_warn_ratio` sourced from `get_settings().budget_warn_ratio`:

- monitor: `_DEFAULT_BUDGET_WARN_RATIO = 0.8` + `_budget_warn_ratio()` (env-only override).
- router: `_BUDGET_WARN_THRESHOLD = 0.9` (hardcoded, no override) — drives the proactive swap.

`AirlockSettings.budget_warn_ratio` already exists (SET-loader), default `0.8`, sourced
`env AIRLOCK_BUDGET_WARN_RATIO > airlock_settings.budget_warn_ratio > default 0.8`.

## Re-grep findings at base f2706e9
- SET-unify did NOT move the monitor's warn-ratio read onto `get_settings()`. The monitor
  still has its own `_DEFAULT_BUDGET_WARN_RATIO = 0.8` and `_budget_warn_ratio()` (reads env
  only, ignores `airlock_settings.budget_warn_ratio`). So BOTH sides need work.
- Router still has `_BUDGET_WARN_THRESHOLD = 0.9` used twice in `_apply_budget_awareness`.

## Decisions (already resolved by the plan)
1. Single ratio: both monitor warn and router swap read `get_settings().budget_warn_ratio`.
   Delete `router._BUDGET_WARN_THRESHOLD` and monitor's `_DEFAULT_BUDGET_WARN_RATIO` /
   `_budget_warn_ratio()`. Keep `settings._DEFAULT_BUDGET_WARN_RATIO` (canonical, DO NOT TOUCH).
2. **Behavior-change #2 (intended):** the router swap point moves from `0.9` to the configured
   ratio (default `0.8`). Recorded in CHANGELOG. Do NOT preserve `0.9`.
3. `0`/absent budget still short-circuits to no-warn/no-swap (AC-0) — unchanged.
4. Client-facing change → harness coverage REQUIRED: add a `dev/smoketest/` warn-ratio
   scenario asserting `X-Airlock-Budget-State: near_limit` fires at the configured ratio
   (ADD, do NOT run — orchestrator runs smoke on a separate dir+port at sign-off).

## Test-isolation fix (SEPARATE commit)
SET-unify added failover catalog filtering (`circuit_breaker` reads
`router.known_model_aliases()`, backed by module-global `router._alias_provider_map`). The
guardian failover tests assume an EMPTY catalog (so the filter is skipped) but don't enforce
it. Fix: autouse fixture in `tests/conftest.py` that calls `set_router_config(None)` before
each test (clears `_alias_provider_map`), making them order-independent.

NOTE on the RED command: the prompt's exact command
`pytest tests/test_fast_router.py tests/test_fast_guardian.py` PASSES at base because
`test_fast_router.py` has a local `_reset_router_config` autouse teardown that clears the
catalog. The order-dependence is nonetheless real: a probe that populates the catalog in a
preceding test (without cleanup) makes both guardian failover tests FAIL with
"skipping failover=… not in model_list catalog". The conftest fixture is the robust global fix.
